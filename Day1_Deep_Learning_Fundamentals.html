§1 — Neural network mechanics

▶ Watch first (intuition) 3Blue1Brown "But what is a neural network?" (Ch.1) · "Gradient descent" (Ch.2)

1.1 What a layer actually computes

A fully-connected layer is an affine map followed by a nonlinearity:

z = W x + b      # affine: linear map + shift
a = φ(z)         # elementwise nonlinearity

An MLP stacks these: a₁ = φ(W₁x+b₁), a₂ = φ(W₂a₁+b₂), … and a final linear head producing logits.

1.2 Why the nonlinearity is non-negotiable

A classic warm-up. Without φ, a deep network collapses to a single linear layer. Compose two linear layers: W₂(W₁x + b₁) + b₂ = (W₂W₁)x + (W₂b₁ + b₂) — just one affine map. Stacking 100 linear layers is still linear: only straight decision boundaries, can't even represent XOR. The nonlinearity is what lets depth buy expressive power (universal approximation intuition).

1.3 Activation functions — what and why

ActivationIntuitionWhy / whenWatch out for
Sigmoid1/(1+e⁻ᶻ) → (0,1)Gates, probabilitiesSaturates → vanishing grad; not zero-centered
Tanh→ (−1,1)Zero-centered sigmoidStill saturates
ReLUmax(0,z)Cheap, no saturation for z>0; default for CNN/MLPDead neurons
GELUz·Φ(z)Smooth; default in transformersSlightly more compute
SwiGLUgated: (W₁x)⊙swish(W₂x)Gating → quality per-param (LLaMA, PaLM)~1.5× FFN params (3 matrices)
  • ReLU killed vanishing gradients on the positive path: its derivative is exactly 1 for z>0 (vs sigmoid's ≤0.25, ~0 when saturated). Stacking sigmoids multiplies many <1 numbers → vanish.
  • Why GELU/SwiGLU in transformers? Smooth + reliable quality bump at scale; SwiGLU's gating modulates information multiplicatively → more expressive per parameter. A "marginal gains compound at scale" story.
⚙ In Practice — dead ReLUs If ReLU units output 0 for every input (so their gradient is 0), they never recover — "dead neurons." See a rising fraction of exact-zero activations in a layer → suspect dead ReLUs from too-aggressive updates (LR too high pushed biases negative, or bad init) → look at the LR and bias gradients. Fixes: lower LR, better init, or GELU/leaky variant.

Self-test §1

Why do neural nets need nonlinearities?▶
Composing affine maps yields an affine map; depth only adds expressive power if a nonlinearity sits between layers. Without it the net is one linear function — can't model non-linear boundaries (e.g. XOR).
Why did ReLU help vs sigmoid?▶
ReLU's gradient is 1 on the active branch, so it doesn't shrink gradients on the forward-active path; sigmoid's derivative is ≤0.25 and ~0 when saturated, so deep stacks multiply many small numbers → vanishing gradient.
Why GELU/SwiGLU instead of ReLU in transformers?▶
Smoothness + empirical quality gains at scale; SwiGLU adds multiplicative gating (more expressive per param). The extra FLOPs/params are justified by better loss-per-compute.

§2 — Backpropagation, from scratch

▶ Watch first (the canonical explanation) 3Blue1Brown Ch.3 "Backprop intuitively" + Ch.4 "Backprop calculus" · Karpathy "Spelled-out intro to backprop" (micrograd) — first ~hour is the derivation.

Backprop is just the chain rule applied efficiently to a computation graph, reusing intermediates so all gradients come from one backward pass. If you can derive it for a 2-layer net, you understand it.

2.1 Setup: a 2-layer MLP for classification

z₁ = W₁ x + b₁
a₁ = φ(z₁)
z₂ = W₂ a₁ + b₂        # logits
ŷ  = softmax(z₂)
L  = cross_entropy(ŷ, y) = −Σ_k y_k log ŷ_k

Goal: compute ∂L/∂W₁, ∂L/∂b₁, ∂L/∂W₂, ∂L/∂b₂.

2.2 The identity that makes it clean: softmax + cross-entropy

A must-know result: the gradient of softmax-cross-entropy w.r.t. the logits is just ŷ − y (predicted probabilities minus one-hot truth).

∂L/∂z₂ = ŷ − y      # call this δ₂, the "error at the output"

Clean and numerically stable (why frameworks fuse the two ops). Why: CE gives ∂L/∂ŷ_k = −y_k/ŷ_k, softmax Jacobian is ŷ_k(δ_ki − ŷ_i); multiply, sum, use Σ y_k = 1 → telescopes to ŷ_i − y_i.

2.3 Propagate backward — the chain rule, layer by layer

δ₂ = ŷ − y
∂L/∂W₂ = δ₂ a₁ᵀ           # outer product: error × the input it multiplied
∂L/∂b₂ = δ₂
∂L/∂a₁ = W₂ᵀ δ₂           # push error back through the linear layer
δ₁     = (W₂ᵀ δ₂) ⊙ φ'(z₁) # gate by local activation derivative
∂L/∂W₁ = δ₁ xᵀ
∂L/∂b₁ = δ₁
★ The whole of backprop, three rules 1. Gradient w.r.t. a weight = (error at its output) × (input it multiplied) → outer product δ aᵀ.
2. Move the error to the previous layer by multiplying by Wᵀ (transpose = sending signal back through the same linear map).
3. Cross a nonlinearity by multiplying elementwise by its local derivative φ'. For ReLU, φ' is 1 where z>0 else 0 — so ReLU just masks the gradient.

2.4 Why vanishing/exploding gradients fall right out

Look at δ₁ = (W₂ᵀ δ₂) ⊙ φ'(z₁). For an L-layer net, the layer-1 gradient is a product of L−1 weight matrices and L−1 activation-derivative factors:

  • Factors consistently < 1 (saturated sigmoids, spectral norm <1) → product shrinks geometrically → vanishing gradient, early layers barely learn.
  • Factors consistently > 1 → product blows up → exploding gradient, NaNs/loss spikes.
★ The unifying lens for all of §3–§5 This single product-of-Jacobians picture motivates almost every trick that follows: careful init (keep factors ~1), normalization (control activation scale), and residuals (provide a "×1" path). Keep it in your head — saying this connection out loud is a strong interview signal.
⚙ In Practice — frameworks fuse softmax+CE Separate softmax then log can overflow (e^large) or underflow (log(0)=−inf). Libraries use a fused, log-sum-exp-stabilized cross_entropy_with_logits. See loss = inf/NaN from step 0, or only when a logit gets large → suspect double-applied softmax (softmax in the model and a loss expecting raw logits) → look at the loss wiring. Extremely common real bug.

Self-test §2

Derive backprop for a 2-layer net.▶
Reproduce §2.2–§2.3. Three rules: weight grad = δ aᵀ; back through linear = Wᵀδ; back through activation = ⊙ φ'.
What's ∂L/∂(logits) for softmax-cross-entropy?▶
ŷ − y. Be ready to sketch why (softmax Jacobian × CE grad telescopes via Σy_k=1).
Explain vanishing/exploding gradients from the equations.▶
Layer-1 gradient is a product of many Wᵀ and φ' factors; consistently <1 → vanish, >1 → explode. Geometric in depth.
Why is ReLU's backward pass essentially a mask?▶
φ'(z)=1 for z>0 else 0 — passes the incoming gradient unchanged or zeros it; no shrinking on the active path.

§3 — Initialization & gradient flow

3.1 The goal of initialization

Keep the variance of activations (forward) and gradients (backward) roughly constant across layers at the start. If each layer multiplies variance by a factor ≠ 1, signal explodes/vanishes before the first useful step.

  • Xavier/Glorot: scale ≈ 1/√fan_in — balances forward/backward variance (linear/tanh regime).
  • He/Kaiming: √(2/fan_in) — the 2 corrects for ReLU killing half the units (halving variance). Default for ReLU.

The principle, not the constants, is the point: init preserves variance layer-to-layer. Get it wrong and you start in a vanishing/exploding regime.

3.2 Why this still matters with normalization

LN/BN reduce init sensitivity, but at the start and at large scale, init still governs early stability. Modern LLMs often scale residual-branch weights by 1/√N_layers so the residual stream variance doesn't grow with depth. Ties back to §2.4 — keep the product-of-Jacobians near 1.

⚙ In Practice — init symptoms See loss huge/NaN at step 0–10 and gradient norms growing with depth → suspect init scale too large → look at init std + missing residual-branch scaling. See loss flat with tiny early-layer grad norms → suspect too-small init (vanishing) → look at per-layer grad-norm logging.

Self-test §3

Why scale init by 1/√fan_in?▶
To keep activation/gradient variance ≈ constant across layers, preventing exploding/vanishing signal at initialization.
Why does He init use √(2/fan_in) for ReLU?▶
ReLU zeros ~half the inputs, halving variance; the factor of 2 compensates.
If normalization exists, why care about init?▶
Init governs the first steps before stats stabilize, and at depth controls residual-stream growth; bad init still causes early NaNs or stalls.

§4 — Normalization & residual connections

4.1 Normalization: what & variants

Normalization layers re-center and re-scale activations so each layer sees a well-conditioned input, stabilizing/speeding optimization, then apply learned γ (scale) and β (shift) so the net can undo it.

VariantNormalizes over…Batch-dependent?Where
BatchNormeach feature, across the batchYesCNNs / vision
LayerNormall features, within one exampleNoTransformers
RMSNormlike LN but no mean-subtraction, RMS onlyNoLLaMA, modern LLMs

4.2 Why transformers use LN/RMSNorm, not BatchNorm

  1. Batch dependence is a liability for sequences. BN stats are noisy with small batches and ill-defined for variable-length / autoregressive decoding (one token at a time). LN normalizes within a single example → independent of batch size and position.
  2. Train/inference mismatch. BN uses batch stats at train, running averages at inference — subtle bugs. LN is identical in both.
  3. RMSNorm drops mean-centering (RMS rescale only): cheaper, ~equal quality — and at LLM scale that compute matters.

4.3 Residual connections — the key trick for depth

A residual block computes y = x + F(x) instead of y = F(x).

  • Gradient highway. ∂y/∂x = I + ∂F/∂x. The identity term gives a gradient path multiplied by 1, not a small weight factor. Re-read §2.4: the + I guarantees one term in the product stays order-1, so gradients flow to early layers even in very deep nets.
  • Easier target. Each block learns a residual (small correction to identity), not a full transform. "Do nothing + small fix" is easier to learn than the identity from scratch.
  • This is why nets went from ~20 to 100s+ of layers, and why every transformer block is wrapped in residuals.

4.4 Pre-norm vs post-norm (frontier-relevant)

  • Post-norm (original): x → Sublayer → Add → LN. Needs careful warmup; unstable at depth.
  • Pre-norm (modern default): x → LN → Sublayer → Add. Residual stream stays clean end-to-end → cleaner gradient highway, far more stable at depth. Almost all large LLMs use pre-norm.
⚙ In Practice — normalization symptoms See stable at batch 256 but diverging at batch 8 → suspect BN's noisy small-batch stats → look at switching to LN/GroupNorm. See a deep post-norm transformer that won't train without long warmup → suspect post-norm instability → consider pre-norm. See eval ≫ worse than train only for BN models → suspect train/running-stat mismatch (left in train mode at eval) → look at model.eval() / BN momentum.

Self-test §4

BatchNorm vs LayerNorm — why transformers prefer LN?▶
BN normalizes per-feature across the batch (batch-dependent, train/inf mismatch, bad for variable-length/decoding); LN within an example (batch-independent, identical train/inf). Transformers process variable-length sequences and decode token-by-token → LN/RMSNorm.
Why do residual connections help training?▶
∂y/∂x = I + ∂F/∂x: the identity gives a gradient path ×1 (prevents vanishing through depth); and each block only learns a correction to identity — an easier target.
Pre-norm vs post-norm — which and why?▶
Pre-norm keeps the residual stream un-normalized (cleaner gradient highway), trains more stably at depth; the modern default for large models.
What does RMSNorm drop vs LayerNorm?▶
Mean-subtraction; it only rescales by RMS. Cheaper, ~equal quality.

§5 — Optimization in practice

5.1 The optimizer ladder (know the why of each step)

  • Batch GD: full dataset per step — accurate but one step per epoch, too slow.
  • SGD / mini-batch: gradient from a small batch — noisy but cheap and frequent; noise helps escape sharp minima. The workhorse.
  • Momentum: velocity v = βv + g, step with v — damps oscillation, accelerates consistent directions. "Heavy ball."
  • RMSProp: divide step by running RMS of recent gradients → per-parameter adaptive LR.
  • Adam: momentum (1st moment) + RMSProp (2nd moment) + bias correction. Robust default.
  • AdamW: Adam + decoupled weight decay. The transformer default (§5.2).

5.2 Why AdamW decouples weight decay (favorite question)

"L2 regularization" and "weight decay" are identical for plain SGD, but not for Adam. In vanilla Adam, adding λw to the gradient means the decay gets divided by the adaptive denominator (√v) — so high-gradient params get less effective decay. That couples regularization to gradient magnitude.

AdamW: w ← w − lr·(adam_update) − lr·λ·w   # decay applied directly, not through √v

Result: uniform pull toward 0, decoupled from adaptive scaling → reliably better generalization for transformers.

5.3 LR schedules & warmup

  • Warmup: start LR ≈ 0, ramp up over first hundreds/thousands of steps. Why: at init, the model and Adam's 2nd-moment estimate v are unreliable; big early steps diverge. Warmup lets stats settle. Especially critical for transformers.
  • Cosine decay (after warmup): anneal LR down on a cosine curve. Large LR early explores; small LR late refines. Standard for LLM pretraining.

5.4 Gradient clipping

Clip the global gradient norm to a threshold (e.g. rescale so ||g|| ≤ 1.0) before the step. Why: occasionally a bad batch / sharp region produces a huge gradient that would throw weights off and spike the loss (the exploding-gradient product from §2.4). Clipping caps the step size — cheap insurance against one bad batch destroying a run.

5.5 Batch size, accumulation, mixed precision, checkpointing

  • Batch size: larger → less noise (stable, often higher LR via sqrt/linear scaling) but diminishing returns & worse exploration; smaller → noisier, sometimes generalizes better. A compute/stability/generalization tradeoff.
  • Gradient accumulation: run several micro-batches, sum gradients, step once → simulates a big batch you can't fit in memory. Gotcha: average correctly; zero grads only after the step.
  • Mixed precision: 16-bit compute for speed/memory, FP32 master copy. BF16 = FP32's exponent range, fewer mantissa bits → rarely overflows, preferred. FP16 = narrow range → often needs loss scaling (scale loss up before backward, unscale before step) to stop small gradients underflowing to 0.
  • Activation (gradient) checkpointing: don't store all activations; recompute during backward. Trades ~30% compute for big memory savings. A memory technique — not numerical stability, and not model checkpoints.
⚙ In Practice — mixed-precision symptoms See loss NaN after enabling FP16 → suspect overflow/underflow in 16-bit → look at loss scaling / switch to BF16. See gradients exactly 0 for small-magnitude params in FP16 → suspect underflow → increase loss scale or use BF16. See fine in FP32 but subtly worse final quality in low precision → suspect precision loss in sensitive ops → keep softmax / LN / loss in FP32.

Self-test §5

Adam vs SGD — when/why?▶
Adam adapts per-parameter LR (momentum + RMS), robust with little tuning, great for sparse/ill-scaled gradients and transformers; SGD(+momentum) can generalize better in some vision settings and is simpler. Default to AdamW for transformers.
Why does AdamW decouple weight decay?▶
In Adam, L2-as-gradient gets divided by the adaptive √v denominator, making decay depend on gradient magnitude. Applying decay directly to weights restores uniform regularization → better generalization.
Why is warmup needed (esp. transformers)?▶
Early on, model and Adam's 2nd-moment estimates are unreliable; big steps then diverge. Ramping LR lets stats settle. Transformers are especially sensitive.
Why clip gradients?▶
To cap the damage from occasional huge gradients (bad batch / sharp region) that would otherwise spike the loss or NaN the run.
How does gradient accumulation simulate a larger batch?▶
Sum gradients over several micro-batches, then take one optimizer step — equivalent to one large batch, trading time for memory.
FP16 vs BF16?▶
BF16 keeps FP32's exponent range (rarely overflows, usually no loss scaling); FP16 has narrow range and typically needs loss scaling to avoid underflow.

§6 — The debugging playbook

The section frontier interviewers love — it tests whether you've actually trained things. Canonical prompt:

Interview prompt"Your loss was decreasing fine, then spikes hard at 30k steps. Walk me through what you check."

No single answer — the skill is an organized differential diagnosis. Say the categories out loud, then prioritize by the clues given (e.g. "spike at a specific, reproducible step on resume → bad data shard or checkpoint/resume bug, not random instability").

6.1 Master symptom → cause → look-at table

6.2 The general method (say this framing)

  1. Localize in time: gradual or sudden? Specific reproducible step (→ data/checkpoint) or random (→ numerical/LR)?
  2. Look at gradient norms first — cheapest, most diagnostic. Spike before the loss spike → exploding/bad batch. ≈0 → vanishing/frozen.
  3. Check the data at that point. Frontier training is data-bound: corrupt shard, tokenization bug, or wrong loss-masking (loss on padding/prompt tokens) are top suspects for localized spikes.
  4. Check the plumbing: double softmax, logits-vs-probs, loss masking, checkpoint/resume of optimizer + scheduler + dataloader, mixed-precision config.
  5. Sanity baseline: can the model overfit a single batch to ~0 loss? If not, something is fundamentally broken — independent of scale. Fastest way to separate "bug" from "needs more compute."
★ The meta-pointYou treat training as an instrumented experiment: log grad norms, activation stats, and data provenance before you need them, and reason about likelihood rather than randomly trying fixes.

Self-test §6

"Loss spikes at 30k steps — what do you check?"▶
Localize (reproducible step? → data shard / resume bug; random? → LR/numerical). Check the grad-norm trace just before, inspect the data batch at that step, verify clipping + mixed-precision config, confirm it's not optimizer-state-on-resume. Give the categories, then prioritize by clues.
"How do you debug NaNs?"▶
Step 0 vs mid-training. Check double-softmax / loss API / init scale first; if FP16-triggered → loss scaling or BF16; scan data for NaN/inf; find where the first NaN appears (forward vs backward).
"Train good, eval bad — causes?"▶
Overfitting, data leakage, train/eval preprocessing mismatch, BN in train mode at eval, or a subtle proxy/goal mismatch (loss improved but capability didn't).
"Bug vs 'needs more compute'?"▶
Try to overfit one batch to ~0. Success → wiring is fine, it's capacity/data/compute; failure → it's a bug (labels, LR, masking, init).

Quick-reference cheat sheet

  • Nonlinearity = the reason depth helps (else net = one affine map).
  • Backprop rules: weight grad = δ aᵀ; back through linear = Wᵀδ; back through activation = ⊙ φ'.
  • Softmax-CE logit gradient = ŷ − y.
  • Vanishing/exploding = product of L Jacobian factors drifting <1 or >1. Everything below fixes this:
    • Init keeps per-layer variance ≈ 1 (1/√fan_in; He √(2/fan_in) for ReLU).
    • Normalization conditions activations (LN/RMSNorm in transformers; not BN — batch dependence + train/inf mismatch).
    • Residuals give a ×1 gradient highway (∂y/∂x = I + ∂F/∂x); pre-norm keeps it clean.
    • Warmup avoids big early steps while Adam's stats are unreliable; cosine decay anneals.
    • Clipping caps damage from occasional huge gradients.
  • AdamW = Adam + decoupled weight decay (decay on weights, not through √v).
  • BF16 > FP16 for range (FP16 needs loss scaling). Grad accumulation simulates big batches. Activation checkpointing trades compute for memory.
  • Debugging reflex: localize in time → grad norms → inspect data → check plumbing (double softmax, loss masking, resume state) → overfit one batch.
