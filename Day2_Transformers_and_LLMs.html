§1 — Tokenization, embeddings & positional encoding

1.1 Tokenization — BPE and why it exists

An LLM can't read text; it reads integer token IDs. Tokenization maps a string ↔ a sequence of IDs from a fixed vocabulary (typically 32k–256k entries). The dominant method is byte-pair encoding (BPE) (and byte-level variants): start from bytes/characters and greedily merge the most frequent adjacent pair, repeatedly, until you reach the target vocab size. Common words become single tokens; rare words split into subword pieces; any string is representable (no out-of-vocab) because the base alphabet is bytes.

The compression tradeoff: a larger vocab → fewer tokens per sentence (shorter sequences → cheaper attention, more text per context) but a bigger embedding + output matrix and rarer tokens seen less often. Vocab size is a real architecture knob.

The vocab-size tradeoff: as vocab grows, tokens-per-sentence falls (cheaper sequences, blue) but the embedding/output matrix size grows linearly (more params + rarer tokens, orange). Real vocabs (32k–256k) sit where the curves balance.

◆ Frontier trick — tokenizer pitfalls that bite real runs
  • Numbers: naive BPE splits "12345" inconsistently → wrecks arithmetic. Fix: split digits individually (Llama) or right-to-left in 3s. A favorite "why is my model bad at math" answer.
  • Code: whitespace/indentation must tokenize efficiently — early GPT-2 wasted tokens on runs of spaces; modern tokenizers add multi-space tokens.
  • Multilingual fairness: a vocab trained on English makes other scripts cost 2–4× more tokens → effectively shorter context and higher price for those users. Vocab composition is a fairness + economics decision.
  • The "SolidGoldMagikarp" class of bug: tokens present in the tokenizer-training corpus but barely in the model-training corpus get garbage embeddings → trigger weird behavior. Glitch tokens are a real eval/robustness concern. (unpacked below)
  • Tokenizer is frozen forever. You pick it before pretraining and can't change it without retraining. Getting it wrong is expensive.
◆ Frontier trick — glitch tokens ("SolidGoldMagikarp"), unpacked

Root cause = two corpora, two stages. The tokenizer is trained first (BPE merges, §1.1) on one corpus; the model is trained later on a different, usually cleaned/reweighted corpus. The vocabulary is frozen after stage 1, so every token gets an embedding row in the model — whether or not the model ever actually sees that token in training.

The failure: some strings are frequent enough in the tokenizer corpus to earn a dedicated merged token, but appear almost never in the model corpus. The canonical " SolidGoldMagikarp" came from a Reddit-username dump present in the tokenizer data but scrubbed from the model's training data.

Why the embedding is garbage: an embedding row is only updated by gradient when its token appears in a batch. A token the model essentially never sees keeps its row near the random initialization — a meaningless vector in a region of embedding space the model never learned to interpret. Feed that token at inference and a random vector enters the residual stream as if it were a real word.

Embedding space (2-D projection). Trained tokens (blue) have been pulled by gradient into meaningful semantic clusters — animals, numbers, code, etc. Glitch tokens (red ✕) were almost never in a training batch, so they never moved: they sit in the diffuse cloud of the random initialization near the origin, in no cluster the model knows how to read.

The detection signal: distance each row moved from its init, $\lVert E_{\text{final}}-E_{\text{init}}\rVert$. Trained tokens travel far (gradient updates accumulate); glitch tokens stay ≈0 — a never-updated row is exactly what a scan for glitch tokens looks for.

Observed symptoms: refusal/evasion ("I can't say that"), hallucinated unrelated words, insults or non-sequiturs, or deterministic-but-bizarre completions. It's the out-of-distribution-input problem localized to one token: in-vocabulary, but out-of-training-distribution.

Why it matters (eval / robustness / security):

  • Eval blind spot: benchmarks never contain these tokens, so the model looks clean while holding thousands of latent landmines.
  • Adversarial handle: someone who knows the glitch tokens can craft inputs that make the model misbehave or bypass guardrails — a cheap jailbreak primitive.
  • Detection: compare tokenizer-frequency vs model-corpus-frequency, or flag embedding rows whose norm / update magnitude is anomalously close to the init distribution (they never moved).
  • Fixes: keep tokenizer and model corpora aligned; prune never-used tokens from the vocab; or explicitly detect untrained tokens and map to UNK / re-initialize / penalize at decode.

The interview-ready one-liner: "Glitch tokens are untrained embeddings — the tokenizer minted a token the model never saw, so its row stayed at random init; at inference it injects noise into the residual stream and the model does something arbitrary. It's an OOD-input bug and an adversarial surface, caught by aligning the two corpora and scanning for never-updated rows."

1.2 Embeddings & weight tying

Each token ID indexes a row of the embedding matrix $E\in\mathbb{R}^{V\times d}$ → a $d$-dim vector. At the output, a projection $\mathbb{R}^{d}\to\mathbb{R}^{V}$ produces logits over the vocab. Weight tying shares one matrix for both ($W_{\text{out}}=E^{\top}$): saves $V\times d$ parameters (large when $V$=128k, $d$=4096 → ~0.5B params), and is well-motivated — the input "what does this token mean" and the output "how likely is this token" live in the same space.

1.3 Positional encoding — and why RoPE won

Attention is permutation-invariant: with no position signal, "dog bites man" = "man bites dog". You must inject order. But why is it permutation-invariant? Unpack it below first — it's the reason positional encoding has to exist at all.

★ Why attention is permutation-invariant (the reason position must be injected)

The core fact: attention reads token content, never position. For token $i$, everything is built from token vectors, with no position index inside:

$$q_i=x_iW_Q,\ \ k_j=x_jW_K,\ \ v_j=x_jW_V,\qquad a_{ij}=\frac{e^{\,q_i\cdot k_j/\sqrt d}}{\sum_{j'}e^{\,q_i\cdot k_{j'}/\sqrt d}},\qquad o_i=\sum_j a_{ij}v_j$$

There is no $i$ or $j$ term anywhere except as a label — so the output is a function of token $i$'s content and the set (bag) of all token contents, not their order:

$$o_i=f\big(x_i,\ \{x_1,x_2,\dots,x_n\}\big)$$

Your example. "Man eat a dog" and "Dog eat a man" hold the same multiset of tokens $\{$man, eat, a, dog$\}$. Take the token dog: in both sentences it attends over the same bag, so its weights $a_{ij}$ and output $o_{\text{dog}}$ are identical. Same for every token → the model computes the exact same representations for both sentences → it cannot tell who is eating whom. Subject-vs-object is pure word-order information, and bare attention discarded it. (They are different training sequences — that's the point: without position the model is blind to the difference and can never learn it.)

The formal statement (permutation equivariance). Let $P$ be a permutation matrix (reorders tokens). Using $P^\top P=I$ and that row-wise softmax commutes with permutation:

$$\text{Attn}(PX)=\text{softmax}\!\big((PXW_Q)(PXW_K)^\top\big)PXW_V=P\,\text{softmax}(QK^\top)V=P\,\text{Attn}(X)$$

Permute the inputs → identical outputs, just permuted the same way. No token's output value changes. (Strictly this is equivariance; "permutation-invariant" is the loose shorthand for "treats the input as a set.")

The fix = positional encoding. Make $x_i$ carry its position so $q,k,v$ become position-dependent: absolute adds $x_i\!\leftarrow\! x_i+p_i$; RoPE rotates $q_i,k_j$ by their positions (giving the relative term $R_{n-m}$, below). Now the bag becomes an ordered sequence and "Man eat a dog" $\ne$ "Dog eat a man" again.

Without position, "Man eat a dog" and "Dog eat a man" are the same multiset of tokens, so attention returns identical per-token outputs (≡). Adding a position to each token breaks the symmetry (≠), restoring word order — and therefore meaning.

You must inject order. Three generations:

  • Sinusoidal / learned absolute (original Transformer): add a position vector to each embedding. Problem: doesn't extrapolate past trained length, and "absolute" is the wrong inductive bias — attention cares about relative distance.
  • Relative position biases (T5, ALiBi): add a bias to attention scores based on $i-j$. ALiBi extrapolates but is coarse.
  • RoPE (Rotary Position Embedding) — the modern default (LLaMA, GPT-NeoX, Qwen, Gemma). Rotate each query/key by an angle proportional to its absolute position; the dot product then depends only on the relative offset.

RoPE rotates 2-D chunks of the $d$-dim vector. For a vector $x$ at position $m$, with per-pair frequency $\theta_k=\text{base}^{-2k/d}$:

$$q_m = R(m)\,q,\quad k_n = R(n)\,k \;\Rightarrow\; q_m^{\top}k_n = q^{\top}R(n-m)\,k$$

The key property: the rotation matrices compose so the score depends only on $n-m$ (relative), yet it's applied with cheap per-position rotations and no added parameters. High-frequency pairs encode short-range order; low-frequency pairs encode long-range — a natural multi-scale clock.

★ RoPE in depth — intuition first

The clock analogy. Think of each pair of dimensions in the query/key vector as the hand of a clock. RoPE rotates that hand by an angle proportional to the token's position. Token at position 0 → no rotation. Position 1 → rotate by $\theta$. Position 5 → rotate by $5\theta$. The information (the hand's length) is untouched; only its angle changes, and the angle encodes "where am I in the sequence."

Why rotation gives you relative position for free. Attention compares a query at position $m$ with a key at position $n$ via a dot product. Two vectors' dot product depends on the angle between them. If you rotate the query by $m\theta$ and the key by $n\theta$, the angle between them changes by $(m-n)\theta$ — it depends only on the difference $m-n$, not on $m$ or $n$ individually. So the model automatically sees "these tokens are 3 apart," and that judgment is the same whether the pair sits at positions (0,3) or (1000,1003). That position-shift-invariance is the whole point — and it's why the same trained weights keep working at positions never seen during training.

Why many different frequencies? One clock hand spinning fast can't tell "2 apart" from "2 + one-full-turn apart" (aliasing). RoPE uses many hands at geometrically decreasing speeds $\theta_k=\text{base}^{-2k/d}$: fast hands resolve fine, local distances; slow hands have wavelengths of thousands of tokens and resolve long-range position. Together they form a positional "fingerprint" at every scale — like how hour/minute/second hands jointly pinpoint a time.

RoPE = rotate each 2-D pair by (position × frequency). Because a dot product depends on the angle between two vectors, rotating query and key by their positions makes the score depend only on the relative offset $m-n$.

The mathematics

2-D core. Take one pair of dimensions $(x_1,x_2)$ and a position $m$. RoPE applies the standard 2-D rotation matrix by angle $m\theta$:

$$R(m\theta)=\begin{pmatrix}\cos m\theta & -\sin m\theta\\ \sin m\theta & \cos m\theta\end{pmatrix},\qquad \tilde x = R(m\theta)\begin{pmatrix}x_1\\x_2\end{pmatrix}$$

Full vector. Split the $d$-dim query/key into $d/2$ consecutive pairs. Pair $k$ gets its own frequency $\theta_k=\text{base}^{-2k/d}$ (base usually $10000$), and is rotated by $m\theta_k$. So the whole rotation is a block-diagonal matrix of these little $2\times2$ rotations:

$$R_m=\mathrm{diag}\big(R(m\theta_0),\,R(m\theta_1),\,\dots,\,R(m\theta_{d/2-1})\big),\qquad q_m=R_m\,q,\quad k_n=R_n\,k$$

The relative-position theorem. Rotation matrices satisfy $R_m^{\top}R_n = R_{n-m}$ (rotating by $-m$ then $+n$ = rotating by $n-m$). Therefore the attention logit is:

$$q_m^{\top}k_n=(R_m q)^{\top}(R_n k)=q^{\top}R_m^{\top}R_n\,k=q^{\top}R_{\,n-m}\,k$$

The absolute positions $m,n$ have cancelled — the score is a function of $q$, $k$, and the offset $n-m$ only. This is the formal statement of "RoPE injects absolute position but attention sees relative position."

Efficient implementation. You never build the matrix. For each pair you precompute $\cos m\theta_k,\sin m\theta_k$ and apply, elementwise:

$$\tilde x_1 = x_1\cos m\theta - x_2\sin m\theta,\qquad \tilde x_2 = x_1\sin m\theta + x_2\cos m\theta$$

# x: [seq, n_heads, head_dim], split into even/odd halves
def apply_rope(x, pos, base=10000.0):
    d = x.shape[-1]
    k = torch.arange(0, d, 2)                  # pair indices 0,2,4,...
    theta = base ** (-k / d)                   # per-pair frequency θ_k
    ang = pos[:, None] * theta[None, :]        # [seq, d/2]  = m·θ_k
    cos, sin = ang.cos(), ang.sin()
    x1, x2 = x[..., 0::2], x[..., 1::2]         # even / odd dims = the 2-D pairs
    out1 = x1 * cos - x2 * sin                  # rotate each pair
    out2 = x1 * sin + x2 * cos
    return interleave(out1, out2)
# applied to q and k ONLY (not v), every layer, before the QKᵀ dot product

Notes that matter in practice: RoPE is applied to Q and K but not V (V carries content, not position); it's re-applied every layer (it's not added once at the input like absolute encodings); and during incremental decoding you must feed each token its true absolute position (the cache offset) — getting this wrong is the Drill 1 bug at the end of this page.

The frequency spectrum: pair index $k$ vs its wavelength $2\pi/\theta_k$ (log scale). Low-$k$ pairs spin fast (short wavelength → resolve nearby positions); high-$k$ pairs spin slowly (wavelengths of thousands of tokens → encode long-range position). A token's full set of rotation angles is a multi-scale positional fingerprint.

◆ Frontier trick — extending context length with RoPE

You trained at 4k but want 128k at inference. You don't retrain from scratch — you rescale RoPE frequencies:

  • Position interpolation (PI): linearly squeeze positions $m\to m\cdot\frac{L_{\text{train}}}{L_{\text{target}}}$ so they fall inside the trained range. Simple, needs a little fine-tuning.
  • NTK-aware / YaRN: change the RoPE base ($\theta$) so high frequencies (local) are preserved while low frequencies (global) stretch — extrapolates far better than uniform PI. YaRN is the production choice for big context extension.
  • Why base matters: a larger base spreads the wavelengths so the longest exceeds the target context, avoiding aliasing. "Did you bump the RoPE theta?" is a real long-context debugging question.

Self-test §1

Why subword tokenization instead of words or characters?▶
Words → huge vocab + OOV; characters → very long sequences (expensive attention) and weak units. Subwords balance: frequent words are one token, rare words decompose, any string is representable from bytes. Vocab size trades sequence length against embedding-matrix size.
Why did RoPE beat absolute positional encodings?▶
It makes the attention score depend on relative position ($n-m$) — the right inductive bias — with no extra parameters and good extrapolation (especially with NTK/YaRN base scaling). Absolute encodings don't extrapolate past trained length and bias toward absolute index.
Your model is bad at multi-digit arithmetic — first thing you check?▶
The tokenizer. Inconsistent digit grouping breaks place value. Check whether digits are split individually / consistently; this is a tokenization problem before it's a modeling one.

§2 — Self-attention, from scratch

2.1 The intuition

Each token builds a query ("what am I looking for?"), and every token offers a key ("what do I contain?") and a value ("what I'll contribute if attended to"). A token's output is a weighted average of all values, weighted by query·key similarity. It's a learned, content-based soft lookup over the sequence — a differentiable dictionary.

2.2 The equation

With $Q,K,V\in\mathbb{R}^{n\times d_k}$ (project the input $X$ by learned $W_Q,W_K,W_V$):

$$\text{Attention}(Q,K,V)=\underbrace{\text{softmax}\!\Big(\frac{QK^{\top}}{\sqrt{d_k}}+M\Big)}_{\text{attention weights }A\in\mathbb{R}^{n\times n}}V$$

$QK^{\top}$ → all pairwise scores ($n\times n$); scale; add mask $M$; softmax each row to a distribution; multiply by $V$ to mix. Every output token is a convex combination of value vectors.

★ What is $V$, and where does it come from?

$V$ is built exactly like $Q$ and $K$ — a linear projection of the same input. Each token's vector $x_j$ (a $d$-dim row) is multiplied by the learned matrix $W_V$:

$$v_j = x_j W_V\in\mathbb{R}^{d_v},\qquad V=XW_V\in\mathbb{R}^{n\times d_v}$$

So $V$ is just the stack of all the value vectors, one row per token. The three projections of the same token play different roles:

  • $q_j$ — "what am I searching for?"
  • $k_j$ — "what do I advertise so others can find me?"
  • $v_j$ — "the actual payload I hand over if someone attends to me."

The output is a weighted average of values: $o_i=\sum_j a_{ij}v_j$. So $Q$ and $K$ only decide the weights $a_{ij}$; $V$ is the information that actually gets mixed and passed to the next layer. Using three separate learned matrices lets "how tokens match" ($Q,K$) be decoupled from "what they contribute" ($V$).

2.3 Why divide by √d_k (must-know)

$q,k$ have components with ~unit variance, so $q\cdot k=\sum_{i=1}^{d_k} q_i k_i$ has variance $\propto d_k$ → std $\propto\sqrt{d_k}$. Without scaling, for large $d_k$ the logits are huge → softmax saturates to one-hot → gradient through softmax $\approx 0$ → attention can't learn. Dividing by $\sqrt{d_k}$ renormalizes the score variance back to ~1, keeping softmax in its responsive regime.

Softmax of scores with std 1 (blue, healthy spread) vs std √64 = 8 (orange, collapsed to a near one-hot spike). The collapsed version has near-zero gradient — the model can't learn what to attend to. Scaling by √d_k keeps you in the blue regime.

★ Why "collapsed → near-zero gradient" (the actual mechanism)

This is softmax saturation — the same idea as a saturated sigmoid/tanh (Day 1). The gradient of softmax outputs $p_i$ w.r.t. its input logits $z_j$ is:

$$\frac{\partial p_i}{\partial z_j}=p_i(\delta_{ij}-p_j)\qquad(\delta_{ij}=1\text{ if }i{=}j,\text{ else }0)$$

When logits are huge (large variance), softmax becomes nearly one-hot: one weight $p_k\approx 1$, the rest $\approx 0$. Substitute:

  • winner: $p_k(1-p_k)\approx 1\cdot 0 = 0$
  • everyone else: $p_i\approx 0$

So every entry of the gradient is ≈ 0 — nudging a score barely changes the (already saturated) weights. And since the scores are $z_{ij}=q_i\cdot k_j$, that zero gradient flows straight back into $Q$ and $K$ (hence into $W_Q,W_K$): the model gets almost no signal about which tokens to attend to, so it can't learn the attention pattern. Dividing by $\sqrt{d_k}$ holds the logit variance near 1, keeping softmax in its responsive region where $p_i(\delta_{ij}-p_j)$ is meaningfully nonzero and gradients flow.

Softmax sensitivity (a proxy for gradient magnitude, $\sum_i p_i(1-p_i)$) vs the logit scale. Near scale 1 (after $\sqrt{d_k}$ scaling) it's high — gradients flow. As the scale grows (unscaled, large $d_k$), it collapses toward 0 — the saturated regime where attention can't learn.

2.4 Causal masking

For autoregressive LMs, token $i$ may not see the future. Set $M_{ij}=-\infty$ for $j>i$ before softmax → those weights become 0. This is what makes the model a valid next-token predictor and lets training compute the loss on all positions in parallel (each position only attends to its past).

The $n\times n$ attention matrix (rows = query token, cols = key token). The upper triangle is masked to $-\infty$ (dark) so no token sees the future; each row of the lower triangle softmaxes to a distribution over its past (brighter = more weight). Row $i$ only mixes values from positions $\le i$.

2.5 Complexity — the central scaling pain

$QK^{\top}$ is $n\times n$: compute and memory are $O(n^2 d)$ in sequence length. Double the context → 4× attention cost. This single fact drives almost all of long-context research.

Attention FLOPs grow quadratically with sequence length while the FFN grows linearly — so beyond some length, attention dominates and context becomes the bottleneck.

◆ Frontier trick — FlashAttention (the answer they want)

Naive attention materializes the full $n\times n$ matrix in HBM → $O(n^2)$ memory, and it's memory-bandwidth-bound, not compute-bound. FlashAttention is an IO-aware exact algorithm: tile $Q,K,V$ into SRAM, compute attention block-by-block with the online-softmax trick (running max + running sum so you never store the full matrix), and never write $A$ to HBM. Result: $O(n)$ memory, 2–4× faster, and it's exact (not an approximation). Key talking points:

  • The win is fewer HBM reads/writes, not fewer FLOPs — attention was bandwidth-bound, so this is the right lever.
  • Online softmax: rescale the running accumulator when a new block reveals a larger max — numerically stable, single pass.
  • Enables the long contexts everyone ships; FlashAttention-2/3 push utilization further (better work partitioning, FP8 on Hopper).
  • Distinguish from approximate methods (Linformer/Performer/sparse) — those trade quality; FlashAttention does not.
⚙ In Practice — softmax numerical stability (the max-subtraction / log-sum-exp trick)

The problem: exp overflows fast — $e^{1000}=\inf$, then $\inf/\inf=$ NaN poisons everything. The fix: subtract the max logit $m=\max_j z_j$ before exponentiating:

$$\text{softmax}(z)_i=\frac{e^{z_i-m}}{\sum_j e^{z_j-m}}=\frac{e^{z_i}}{\sum_j e^{z_j}}\qquad(\text{the }e^{-m}\text{ cancels — same shift-invariance as z-loss})$$

Mathematically identical, but now the largest exponent is $e^{0}=1$ → nothing overflows; only genuinely-negligible terms round to 0. Same for the log-partition: $\log Z=m+\log\sum_j e^{z_j-m}$ (the log-sum-exp trick).

Worked example $z=[1000,1001,1002]$: naïve → $e^{1000}=\inf$ → NaN 💥. Subtract max 1002 → shifted $[-2,-1,0]$ → $e=[0.135,0.368,1.0]$ → softmax $[0.090,0.245,0.665]$ ✓.

Notes: every sane library does this inside softmax/log_softmax/cross_entropy — a hand-rolled exp(z)/sum is the unsafe version. FlashAttention (§2.5) generalizes it to streaming via a running max (online softmax). And it can't replace z-loss: max-subtraction is a forward-pass reparametrization invisible to gradients, so it doesn't stop logit drift. If you see NaNs localized to attention, suspect an un-maxed softmax or a fully-masked row (all $-\infty\to 0/0$).

Self-test §2

Walk through attention shapes end-to-end.▶
$X\in\mathbb{R}^{n\times d}$ → $Q,K,V=XW_{Q,K,V}\in\mathbb{R}^{n\times d_k}$. $QK^\top\in\mathbb{R}^{n\times n}$ → scale by $1/\sqrt{d_k}$, add causal mask, row-softmax → $A\in\mathbb{R}^{n\times n}$ → $AV\in\mathbb{R}^{n\times d_k}$. Output is a per-token convex combination of values.
Why $\sqrt{d_k}$ and not $d_k$ or nothing?▶
Dot product over $d_k$ unit-variance terms has variance $\propto d_k$, std $\propto\sqrt{d_k}$. Dividing by $\sqrt{d_k}$ restores unit-variance logits so softmax doesn't saturate (→ vanishing gradient). $d_k$ would over-shrink; nothing → collapse.
What does FlashAttention actually optimize?▶
HBM memory traffic, not FLOPs. It tiles into SRAM and uses online softmax to avoid materializing the $n\times n$ matrix → $O(n)$ memory, 2–4× faster, exact. Attention was bandwidth-bound, so reducing IO is the right lever.
Why is long context expensive even with FlashAttention?▶
FlashAttention fixes memory ($O(n)$) but compute is still $O(n^2 d)$. And at inference the KV cache grows linearly with context and dominates memory/bandwidth (→ §4).

§3 — Multi-head attention & the full block

3.1 Multi-head attention (MHA)

One attention is one "relationship." Split $d$ into $h$ heads of size $d_k=d/h$, run attention independently per head, concatenate, project:

$$\text{MHA}(X)=\big[\text{head}_1;\dots;\text{head}_h\big]W_O,\qquad \text{head}_i=\text{Attention}(XW_Q^i,XW_K^i,XW_V^i)$$

Heads specialize (syntax, coreference, induction/copy heads, positional). Same total compute as one big head, but multiple subspaces → richer relational modeling.

3.2 GQA / MQA — the KV-cache-driven redesign

At inference, you cache one K and one V per head per token. MHA with many heads makes that cache enormous (→ §4). Multi-Query Attention (MQA) uses many query heads but one shared K/V head. Grouped-Query Attention (GQA) is the middle ground: groups of query heads share a K/V head (e.g. 8 KV heads for 64 Q heads). It's the production default (Llama 2/3 70B, Mistral) because it shrinks the KV cache ~8× with negligible quality loss.

Relative KV-cache size: MHA (64 KV heads) vs GQA-8 vs MQA-1 (log axis). GQA captures almost all the memory savings of MQA while keeping more quality than the single-KV-head extreme.

3.3 The full decoder block (modern / pre-norm)

Each block is two residual sub-layers with pre-norm (Day 1 §4.4):

x = x + attn(RMSNorm(x))      # attention sublayer

◆ Frontier trick — the modern-architecture "why" list
  • SwiGLU: gated activation empirically beats ReLU/GELU FFN at equal compute. Note the size convention: to keep params equal, the hidden dim is scaled by $\frac{2}{3}$ (SwiGLU has 3 matrices, not 2) → you'll see FFN dims like $\frac{8}{3}d$.
  • QK-norm: apply RMSNorm to Q and K before attention → stabilizes logits at scale, prevents attention-logit blow-up (a known large-model instability). Used in several recent frontier models.
  • No bias terms: biases add instability and ~nothing to quality at scale; dropping them is cleaner and slightly cheaper.
  • z-loss: a small auxiliary penalty on the log-partition of the output softmax keeps logits from drifting large → numerical stability in bf16. (unpacked below)
  • Parallel attn+FFN (GPT-J/PaLM style): run both sublayers off the same norm'd input to cut latency; small quality hit, sometimes used.
◆ Frontier trick — z-loss, unpacked

The partition function. Softmax over logits $z=(z_1,\dots,z_V)$ is $p_i=e^{z_i}/Z$ with $Z=\sum_j e^{z_j}$ — the partition function (the normalizer). Its log, $\log Z=\log\sum_j e^{z_j}=\text{logsumexp}(z)$, is the log-partition.

Root cause — softmax is shift-invariant. Add the same constant $c$ to every logit and nothing changes:

$$\text{softmax}(z+c)_i=\frac{e^{z_i+c}}{\sum_j e^{z_j+c}}=\frac{e^{c}e^{z_i}}{e^{c}\sum_j e^{z_j}}=\text{softmax}(z)_i$$

So cross-entropy depends only on the logit differences, never their absolute magnitude. Nothing in the objective pins the overall scale → over a long run the logits (and $\log Z$) can drift larger and larger without the loss objecting.

Why that's dangerous (esp. bf16). Large logits → $e^{\text{large}}$ → overflow/inf or precision loss in bf16's ~2–3 significant digits (Day 1 §5.5) → loss spikes / instability, and mis-calibrated outputs. The model has no incentive to keep logits small, so at scale they creep up — right where bf16 is least forgiving.

The fix. Add a tiny auxiliary penalty on the log-partition, pulling it toward 0:

$$\mathcal{L}_z=\lambda\,(\log Z)^2=\lambda\Big(\log\textstyle\sum_j e^{z_j}\Big)^2,\qquad \lambda\approx 10^{-4}$$

Squaring penalizes $\log Z\neq 0$ either way → drives $Z\to 1$ → pins the logits' absolute scale to a bounded range, removing the shift-freedom the model was abusing. $\lambda$ is small enough to not distort the real objective.

Two places you'll see it: (1) output z-loss on the vocabulary softmax (PaLM); (2) router z-loss in MoE models (Switch / ST-MoE), where the expert-router softmax is even more spike-prone.

Max logit magnitude over training: without z-loss (orange) the scale drifts upward unchecked (softmax doesn't care) until it risks bf16 overflow; with z-loss (blue) the $\lambda(\log Z)^2$ term keeps it bounded near a stable value.

How it assembles into the final loss — a weighted sum with cross-entropy, per token:

$$\mathcal{L}_{\text{total}}=\underbrace{-\tfrac1N\sum_t \log p_\theta(x_t\mid x_{\lt t})}_{\mathcal{L}_{\text{CE}}}+\;\lambda\cdot\tfrac1N\sum_t(\log Z_t)^2$$

It's nearly free because CE already needs the log-partition: $\mathcal{L}_{\text{CE},t}=\log Z_t-z_{t,y_t}$ (log-partition minus the correct-class logit). So z-loss just squares a scalar you already computed — no extra matmul.

logz = torch.logsumexp(logits, dim=-1)          # [N] — already needed for CE
ce   = logz - logits.gather(-1, y[...,None]).squeeze(-1)
loss = (ce + lam * logz**2)[mask].mean()        # mask padding; same tokens as CE

In distributed training:

  • Data parallel / FSDP: nothing special — a per-token scalar added before the usual gradient all-reduce; just average over the global non-masked token count, like CE.
  • Tensor / vocab parallelism (the interesting case): logits are sharded across the vocab dim, so computing softmax/CE already does an all-reduce of the max and sum-of-exp across shards to get the global $\log Z$. z-loss = $(\log Z)^2$ piggybacks on that exact reduction → essentially free, which is why Megatron/PaLM fold it into the vocab-parallel CE kernel.
  • Pipeline parallel: loss is on the last stage only; no special handling. Gotcha: apply z-loss on the same non-masked tokens as CE or the terms get mis-weighted.

Is it worth it? ~Zero compute, one hyperparameter. Real spike/instability reduction at frontier scale in bf16, plus better calibration. Most valuable for very large models, bf16/fp8, and especially MoE routers (router z-loss is near-mandatory there). For a small fp32 model with other stabilizers it's marginal — but since it's free, people leave it in.

"Can I just normalize logits to the mean instead?" — No. Subtracting the per-token mean (or max) is invisible to training because softmax is shift-invariant: $\text{softmax}(z-\bar z)=\text{softmax}(z)$ → same loss, same gradients, same drift. It's only a forward-pass numerical trick (you should already subtract the max for safe exp) — it fixes immediate overflow in the computation but adds no gradient pressure to keep the weights' logits small, so the drift, the bf16 precision loss, and the spikes remain. z-loss is different in kind: it injects a real gradient that pushes the weights toward a bounded $\log Z$.

Alternatives that do work (they scale or cap — not just shift): logit soft-capping (Gemma 2: $z\leftarrow c\cdot\tanh(z/c)$ on attention & final logits, hard-bounds magnitude); QK-norm (fixes the attention-logit blowup specifically); final-layer norm / output-head weight decay (weaker, indirect). Mean/max subtraction is necessary for numerical safety but useless as a drift constraint.

Should I apply z-loss to every softmax (attention, output, …)? — No. Apply it only where a softmax produces a decision/output distribution whose absolute logit scale is unconstrained and drift is harmful, and where $\log Z$ is already computed:

Softmaxz-loss?Why

Why not the attention softmax: (1) it's an intermediate, not a loss — z-loss'ing it adds a penalty for every query × head × layer × position, thousands of auxiliary terms fighting the main objective; (2) the attention pathology (logit blow-up / entropy collapse) is fixed more cleanly by controlling the input scale — QK-norm bounds $q\!\cdot\!k$ directly, or Gemma-2 soft-caps it; (3) pushing $\sum_j e^{q_i\cdot k_j}\to 1$ over $n$ keys would shove logits very negative and interact badly with value-averaging. So: z-loss → output head + router; attention → QK-norm / cap. Auxiliary losses should be minimal and targeted.

"No extra matmul" — but isn't there backward compute? Yes, a little — but it's elementwise and reuses tensors CE already made, so it's negligible. The gradient of the log-partition is the softmax (a classic identity): $\frac{\partial \log Z}{\partial z_j}=p_j$, so $\frac{\partial \mathcal{L}_z}{\partial z_j}=2\lambda\log Z\cdot p_j$. Combined with CE ($\frac{\partial \mathcal{L}_{\text{CE}}}{\partial z_j}=p_j-\mathbb{1}[j{=}y]$), the total gradient into the logits is:

$$\frac{\partial \mathcal{L}_{\text{total}}}{\partial z_j}=p_j\big(1+2\lambda\log Z\big)-\mathbb{1}[j{=}y]$$

Same shape as the CE gradient — just one extra scalar $(1+2\lambda\log Z)$ rescaling the already-computed $p_j$. So the extra work is: forward = square a per-token scalar ($O(N)$); backward = one broadcast scalar-multiply into the $[N,V]$ gradient you were already forming ($O(NV)$ elementwise). The matmuls (hidden→logits and its backward) are unchanged — z-loss's gradient flows into the same dL/dlogits tensor that goes through the same backward matmul, and $\log Z$, $p$ are already retained. "Free" = matmul-free + reuses existing tensors (overhead <~1%), not literally zero FLOPs.

★ Recap — the one quantity that governs gradient explosion / vanishing

Backprop multiplies a Jacobian at every layer, so the gradient reaching layer $\ell$ is a product of per-layer Jacobians:

$$\frac{\partial \mathcal{L}}{\partial h_\ell}=\frac{\partial \mathcal{L}}{\partial h_L}\prod_{k=\ell+1}^{L} J_k,\qquad J_k=\frac{\partial h_k}{\partial h_{k-1}}$$

If each layer scales the gradient's magnitude by a factor $\gamma$ — the spectral radius (largest singular value) of $J_k$ — then across depth the magnitude goes as:

$$\Big\|\tfrac{\partial \mathcal{L}}{\partial h_\ell}\Big\|\approx \gamma^{\,L-\ell}\Big\|\tfrac{\partial \mathcal{L}}{\partial h_L}\Big\|\qquad\Rightarrow\qquad \gamma>1\ \text{explode},\ \ \gamma<1\ \text{vanish},\ \ \gamma\approx1\ \text{stable}$$

So the single governing quantity is the per-layer gain $\gamma$, and the failure is exponential in depth. $\gamma$ is set by the weight scale (init), the activation derivative, and (in RNNs) the recurrent eigenvalue. Every stability technique is a way to pin $\gamma\approx 1$:

  • Init (Xavier/He) — set the initial weight scale so $\gamma\approx1$ at step 0.
  • Normalization (RMSNorm/LayerNorm) — rescale activations each layer → bound $J_k$'s scale.
  • Residual connections — $\partial y/\partial x=I+\partial F/\partial x$; the $+I$ guarantees an order-1 gradient path ($\gamma\approx1$). This is the reason deep transformers train.
  • Pre-norm — keep the residual highway clean so that $+I$ path isn't disrupted (Day 1 §4.4).
  • Gradient clipping — cap the rare batch where $\gamma^L$ momentarily spikes.
  • Transformer-specific: $\sqrt{d_k}$ (keeps softmax responsive so gradients flow, §2.3), QK-norm / z-loss / soft-cap (keep logits bounded so $J_k$ stays well-behaved).

Gradient magnitude vs depth for three per-layer gains. A 10% drift either way ($\gamma=1.1$ vs $0.9$) becomes a ~10²× explosion or collapse over 50 layers — why deep nets are so sensitive to $\gamma$, and why residuals + norm + good init exist to hold $\gamma\approx1$. (Full derivation: Day 1 §2.4 / §6.1b.)

3.4 Self-attention vs cross-attention (and Flamingo)

Same equation — $\text{softmax}(QK^\top/\sqrt d)V$ — but the difference is where Q, K, V come from:

  • Self-attention: $Q$, $K$, $V$ all come from the same sequence. Every token looks at the other tokens in its own sequence. This is what a decoder block uses (with a causal mask).
  • Cross-attention: $Q$ comes from one sequence (the "reader"), but $K$ and $V$ come from a different sequence (the "source"). The reader pulls information out of the source. No causal mask on the source — you can see all of it.

The classic home of cross-attention is the encoder–decoder Transformer (translation): the decoder self-attends over the text generated so far, then cross-attends into the encoder's representation of the source sentence. It's also exactly the mechanism a VLM uses to let text tokens read visual tokens.

Self vs cross attention is the same math; only the source of K/V changes. In cross-attention a reader token's query scans an entirely separate set of keys/values — the encoder's tokens, or an image's visual tokens.

◆ Frontier trick — Flamingo's gated cross-attention to visual tokens

The problem Flamingo solved: you have a strong, frozen pretrained LLM and a frozen vision encoder, and you want the LLM to see images without retraining it (which would be expensive and could damage its language ability). You can't just prepend thousands of image tokens into the context cheaply, and you want to protect the LLM's weights.

The solution: keep the LLM blocks frozen and insert new cross-attention layers between them. The text tokens (queries) cross-attend into the visual tokens (keys/values) produced by the vision encoder + a Perceiver Resampler (which compresses a variable number of image patches into a small fixed set of tokens). Only these new layers + resampler are trained.

The "gated" part (the clever bit): each inserted cross-attention (and its FFN) is wrapped with a tanh gate initialized to 0:

$$x \leftarrow x + \tanh(\alpha)\cdot \text{crossattn}(\;q{=}x,\ kv{=}\text{visual tokens}\;),\qquad \alpha \text{ init } 0$$

At init $\tanh(0)=0$, so the new layer contributes nothing → the model is exactly the original frozen LLM on day one (no shock, no degradation). As training proceeds, $\alpha$ grows and the model gradually learns how much visual information to let in. This is the same "start as identity, ease in" idea as residual/zero-init tricks (Day 1), applied to bolt vision onto a frozen LLM.

Why this matters / when it's used: it's the cross-attention school of VLMs (Flamingo, IDEFICS) — vision enters through dedicated cross-attn layers, leaving the text stream and KV cache mostly intact. The competing self-attention school (LLaVA, most modern VLMs) instead projects visual tokens into the embedding space and just concatenates them into the input sequence, so ordinary self-attention mixes text and image together. Trade-off: cross-attention keeps the LLM frozen and decouples image-token count from sequence length (cheaper context), but adds parameters and is a bit less flexible; concatenation is simpler and unifies the modalities but pays full sequence-length cost for every visual token.

Flamingo: a frozen vision encoder + Perceiver Resampler emit a small fixed set of visual tokens; new gated cross-attention layers (trained, purple) are interleaved into the frozen LLM (blue/🔒). Text tokens (Q) read visual tokens (K,V). The $\tanh(\alpha)$ gate starts at 0 so the LLM is unchanged at init and eases vision in during training.

The gate $\tanh(\alpha)$ over training: starts at 0 (cross-attention is a no-op → LLM behaves exactly as before), then grows as the model learns how much visual signal to admit. Zero-init gating = "do no harm on day one," the same identity-start principle behind residual connections.

3.5 Attention variants used at the frontier (and why they win)

Plain MHA has two pain points (Day 2 §2.5, §4): compute is $O(n^2)$ and the KV cache grows linearly with context, dominating inference memory/bandwidth. Every frontier variant attacks one or both. Two orthogonal axes: (a) shrink the KV cache (MQA → GQA → MLA), and (b) avoid the $n^2$ / the growing cache entirely (sliding-window, sparse, linear/SSM).

Most variants are just different masks on the $n\times n$ score matrix — they change which (query, key) pairs are allowed. Rows = query position, columns = key position; a filled cell = "this query attends to this key."

Left→right: causal (dense lower triangle, $O(n^2)$) · sliding-window (only the last $w$ keys per query → a band, $O(nw)$) · dilated/strided (every $k$-th key → long reach, few cells) · global+local (a band plus a few "global" tokens every query sees — BigBird/Longformer style). Fewer filled cells = less compute and smaller cache.

MLA — Multi-head Latent Attention (DeepSeek)

The idea: GQA/MQA shrink the cache by throwing away K/V heads (less expressive). MLA instead keeps full per-head expressiveness but stores a low-rank compressed latent and reconstructs K,V on the fly. For each token, project the hidden state down to a small latent $c_t$ and cache only that:

$$c_t = W_{DKV}\,h_t\in\mathbb{R}^{d_c}\ (d_c\ll d),\qquad K_t = W_{UK}\,c_t,\quad V_t = W_{UV}\,c_t$$

You cache $c_t$ (e.g. $d_c\approx512$) instead of full K and V across all heads — a much smaller footprint than even GQA — yet the up-projections $W_{UK},W_{UV}$ give each head its own K,V, so quality stays at ~MHA level. (RoPE needs a small decoupled extra key dimension, since you can't rotate a position-free latent — a known MLA detail.) Why it wins: it pushes the cache-size-vs-quality frontier below GQA, which is why DeepSeek-V2/V3 can serve very long context cheaply.

Sliding-window attention + attention sinks (Mistral, Gemma)

The idea: each token attends only to the previous $w$ tokens (e.g. 4096), not the whole history → compute drops to $O(n\,w)$ and the KV cache becomes a fixed-size rolling buffer (bounded memory regardless of total length). The catch — "won't it lose long-range info?" — is answered by depth: like a CNN's receptive field, stacking $L$ windowed layers lets information propagate up to $\approx L\times w$ tokens away, because each layer can relay what it saw to the next.

Attention sinks (StreamingLLM): naive sliding windows blow up when you evict the very first tokens, because the model dumps excess attention probability onto those first positions (a learned "sink"). Keeping the first few tokens permanently in the window stabilizes streaming generation to unbounded length. "Did you keep the sink tokens?" is a real long-context inference question.

Linear attention & state-space models (Mamba, RWKV; hybrids like Jamba)

The idea: softmax attention is what forces $O(n^2)$ and a growing KV cache. Replace it with either a kernel feature map (linear attention: $\text{softmax}(QK^\top)V \to \phi(Q)\,(\phi(K)^\top V)$, reorderable to $O(n)$) or a selective state-space model (SSM) that processes the sequence as a recurrence with a fixed-size hidden state. Decoding then needs constant memory — no cache that grows with context.

$$\text{SSM:}\quad s_t = A\,s_{t-1} + B\,x_t,\qquad y_t = C\,s_t\qquad(\text{state } s_t \text{ is fixed size, independent of } t)$$

Why it wins / the catch: linear time + $O(1)$ decode state make ultra-long context cheap. But a fixed-size state can't store everything → these models are weaker at exact recall (the "needle-in-a-haystack" retrieval that full attention nails). So the frontier answer is usually a hybrid (e.g. Jamba): interleave a few full-attention layers among many Mamba/linear layers — keep exact-recall ability where it matters, pay $O(n^2)$ only on a minority of layers.

Cost vs sequence length: full attention $O(n^2)$ (orange) explodes; sliding-window $O(n\,w)$ (blue) and linear/SSM $O(n)$ (green) stay affordable at long context. This is the whole motivation for the variants above.

Per-token KV-cache footprint (log scale): MHA → GQA (~8×) → MQA (~64×) shrink by dropping K/V heads; MLA goes further via a compressed latent while keeping ~MHA quality; linear/SSM have a constant state (no per-token cache at all). Smaller cache = longer context + bigger batch at decode (Day 2 §4).

◆ Frontier trick — how to talk about attention variants
  • Name the two axes: "shrink the KV cache" (MQA/GQA/MLA — same $O(n^2)$ compute, cheaper memory) vs "kill the quadratic / the growing cache" (sliding-window, sparse, SSM). Conflating them is a tell.
  • MLA > GQA framing: GQA saves memory by removing heads (expressiveness cost); MLA saves memory by compressing then reconstructing, so it keeps MHA-like quality at a smaller cache — a strictly better point on the frontier.
  • FlashAttention is orthogonal: it's an exact, IO-aware implementation (Day 2 §2.5), not a different attention. It composes with causal/sliding/GQA. Don't list it as a "variant."
  • Why hybrids, not pure SSM: fixed state ⇒ lossy recall; a few interleaved full-attention layers restore needle-in-haystack while keeping most layers linear. This is the current frontier consensus for long context.
  • Sliding-window's secret: depth gives global reach (receptive field $\approx L\times w$) — and remember attention sinks for stable streaming.

§4 — Inference: KV cache, prefill vs decode, long context

4.1 The KV cache — why it exists

Generation is autoregressive: one token at a time. Without caching, generating token $t$ would recompute K and V for all previous $t-1$ tokens every step → $O(n^2)$ wasted work. The KV cache stores each layer's K and V for all past tokens, so each new step only computes Q,K,V for the one new token and attends against the cache. This turns per-step cost from $O(n)$ recompute into $O(1)$ new work (plus the attention read over the cache).

Cache size is the dominant inference memory cost:

$$\text{KV bytes}=2\;(\text{K and V})\times L_{\text{layers}}\times n_{\text{ctx}}\times h_{kv}\times d_{\text{head}}\times \text{bytes/elt}$$

This is per sequence and grows linearly with context — which is exactly why GQA (shrinks $h_{kv}$) and KV quantization (shrinks bytes/elt) matter so much.

Prefill computes K,V for every prompt token in one parallel pass (compute-bound) and stores them. Each decode step then computes K,V for just the one new token, appends it (purple), and attends over the entire cache — so the per-step cost is dominated by reading the ever-growing cache (bandwidth-bound).

Inference has two phases with opposite bottlenecks — getting this distinction crisp is a strong signal:

Roofline view: decode (low arithmetic intensity) sits on the memory-bandwidth slope — adding FLOPs doesn't help; you must raise intensity (batch) or cut bytes (GQA, quantized KV). Prefill sits near the compute roof.

4.3 Long context ties it all together

Two costs grow with context: attention compute ($O(n^2)$, §2.5) and the KV cache ($O(n)$ memory + bandwidth). At long context, decode becomes dominated by reading the ever-growing KV cache every single token.

KV-cache memory vs context length for MHA vs GQA-8 (illustrative). At 128k tokens the MHA cache for a single sequence can rival the model weights — GQA is what makes long context affordable.

◆ Frontier trick — the inference-systems toolkit
  • Continuous (in-flight) batching: don't wait for a batch to finish — swap finished sequences out and new ones in every step. The core throughput win in vLLM/TGI.
  • PagedAttention: store the KV cache in non-contiguous "pages" (like OS virtual memory) to eliminate fragmentation and enable prefix sharing. Lets you pack far more concurrent sequences.
  • Speculative decoding: a small draft model proposes $k$ tokens; the big model verifies them in one parallel forward and accepts the longest correct prefix. 2–3× decode speedup with identical output distribution.
  • Chunked prefill: interleave prefill chunks with decode steps so a long prompt doesn't stall everyone's token generation (latency fairness).
  • KV quantization / GQA / sliding-window + sinks: shrink the cache (bytes/elt, heads, or length kept). "Attention sinks" (keep the first few tokens) make sliding-window long-gen stable.
  • MFU vs MBU: talk in Model-FLOPs-Utilization for prefill/training and Model-Bandwidth-Utilization for decode — naming the right efficiency metric per phase signals real experience.

Avoids recomputing K/V for all past tokens each decode step ($O(n^2)\to O(n)$ work). Cost: memory $=2\cdot L\cdot n\cdot h_{kv}\cdot d_{\text{head}}\cdot\text{bytes}$ per sequence, growing linearly with context → dominates inference memory and decode bandwidth.
Prefill vs decode — which is compute- vs memory-bound, and why?▶
Prefill is compute-bound (big parallel matmuls over the whole prompt). Decode is memory-bandwidth-bound (one token wide, must stream all weights + KV from HBM per step → tiny arithmetic intensity). So batching helps decode most.
How does speculative decoding speed up inference without changing outputs?▶
A cheap draft model proposes several tokens; the target verifies them in one parallel pass and accepts the longest matching prefix (rejection sampling preserves the target's exact distribution). Turns several memory-bound steps into one → 2–3×.
Why is batching the key decode lever but not the key prefill lever?▶
Decode is bandwidth-bound; batching amortizes the per-step weight load across sequences, raising arithmetic intensity. Prefill is already compute-bound and parallel, so batching helps less there.

§5 — Pretraining: objective, perplexity, scaling laws

5.1 Next-token prediction = maximum likelihood

The pretraining objective is just MLE on text: maximize the probability the model assigns to the real next token, i.e. minimize average cross-entropy / negative log-likelihood:

$$\mathcal{L}=-\frac1N\sum_{t=1}^{N}\log p_\theta\!\left(x_t \mid x_{\lt t}\right)$$

Per position it's cross-entropy between the one-hot true token and the model's softmax. Causal masking lets you compute this for every position in a sequence in one forward pass.

5.2 Perplexity

Perplexity is just the exponentiated loss — the model's "effective branching factor" (how many equally-likely tokens it's choosing among):

$$\text{PPL}=\exp(\mathcal{L})=\exp\!\Big(-\tfrac1N\textstyle\sum_t\log p_\theta(x_t\mid x_{\lt t})\Big)$$

PPL = 1 → perfect; PPL = $V$ → uniform guessing. It's only comparable across models with the same tokenizer (different tokenization changes $N$ and the units), which is a common eval gotcha.

5.3 Scaling laws & Chinchilla (frontier-critical)

Loss falls as a smooth power law in parameters $N$, data $D$, and compute $C$. The Chinchilla form:

$$L(N,D)=E+\frac{A}{N^{\alpha}}+\frac{B}{D^{\beta}}$$

$E$ = irreducible entropy of language; the two terms are finite-model and finite-data error. Given a fixed compute budget $C\approx 6ND$, there's an optimal split. Chinchilla's finding: earlier models (GPT-3) were too big, trained on too little data. Optimal scaling grows $N$ and $D$ together — roughly ~20 tokens per parameter.

IsoFLOP intuition: at a fixed compute budget, loss vs model size is U-shaped — too small underfits, too big sees too little data. The minimum is the compute-optimal point; connecting minima across budgets gives the scaling frontier.

The other view: along the compute-optimal frontier, loss falls as a power law in compute — a straight line on log-log axes ($L-E\propto C^{-p}$). Smooth and predictable over many orders of magnitude, which is what lets labs forecast a big run's loss from small ones. It bends toward the irreducible floor $E$ only asymptotically.

◆ Frontier trick — what Chinchilla actually implies in practice
  • Compute-optimal ≠ deployment-optimal. Chinchilla minimizes training loss per FLOP. If you'll serve the model to trillions of tokens, it's worth over-training a smaller model past 20 tok/param (Llama uses 200×+) so inference is cheaper forever. This "inference-aware scaling" trade signals real deployment thinking.
  • $C\approx 6ND$: memorize this — 6 FLOPs per parameter per token (2 forward + 4 backward). Lets you sanity-check any training budget on a napkin.
  • Data wall: at frontier scale you run out of high-quality unique tokens. Then: repeat data (a few epochs is fine, returns diminish), synthetic data, or shift compute into post-training/test-time. Scaling-law extrapolation assumes fresh tokens.
  • Batch size & LR scale too. There's a critical batch size; LR follows width-based heuristics. μP (maximal-update parameterization) lets you tune hyperparameters on a small model and transfer to the large one — a real cost-saver.
  • "Emergent abilities" caveat: some apparent sharp emergence is a metric artifact (discontinuous metrics); smooth log-likelihood gains can look like jumps. Have a nuanced take.

§6 — Post-training: SFT → RLHF → DPO

Pretraining yields a clever next-token predictor that isn't an assistant. Post-training aligns it to be helpful, harmless, and instruction-following.

6.1 Supervised fine-tuning (SFT)

Fine-tune on curated (prompt → ideal response) pairs with the same next-token loss, but mask the loss on the prompt — you only train on the response tokens (and never on padding). This teaches format and instruction-following. SFT data quality > quantity (a few thousand great examples beat millions of mediocre ones — the "LIMA" lesson).

6.2 Why RL at all? (the framing to nail)

SFT can only imitate demonstrations. For many qualities (helpfulness, harmlessness, "which of two answers is better") it's far easier for humans to compare outputs than to write the ideal one. RL lets you optimize a preference signal over the model's own samples — pushing up good behaviors and down bad ones, beyond what any fixed demonstration set encodes.

6.3 RLHF — the classic pipeline

  1. Collect preferences: sample pairs of responses, humans pick the better → a dataset of $(x, y_w, y_l)$.
  2. Train a reward model $r_\phi$ (Bradley–Terry): $\;\mathcal{L}_{RM}=-\log\sigma\big(r_\phi(x,y_w)-r_\phi(x,y_l)\big)$.
  3. Optimize the policy with PPO against the reward, with a KL penalty to the SFT model so it doesn't drift into gibberish that games the reward:

$$\max_{\pi_\theta}\ \mathbb{E}_{x,\,y\sim\pi_\theta}\big[r_\phi(x,y)\big]\;-\;\beta\,\mathrm{KL}\!\big(\pi_\theta(\cdot\mid x)\,\Vert\,\pi_{\text{ref}}(\cdot\mid x)\big)$$

The KL term is the safety leash: $\beta$ too small → reward hacking (degenerate high-reward text); too large → the policy never improves. Tuning $\beta$ is a core RLHF skill.

RLHF threads three stages (SFT → reward model → PPO with a KL leash). DPO proves the optimum of that KL-regularized objective has a closed form, collapsing the reward-model + RL stages into a single offline logistic loss against the frozen SFT reference — simpler and cheaper, at the cost of online exploration.

6.4 DPO — skip the reward model

Direct Preference Optimization shows the RLHF objective has a closed-form optimum, letting you optimize preferences with a simple classification loss directly on the policy — no separate reward model, no online RL loop:

$$\mathcal{L}_{DPO}=-\mathbb{E}_{(x,y_w,y_l)}\log\sigma\!\Big(\beta\log\tfrac{\pi_\theta(y_w\mid x)}{\pi_{\text{ref}}(y_w\mid x)}-\beta\log\tfrac{\pi_\theta(y_l\mid x)}{\pi_{\text{ref}}(y_l\mid x)}\Big)$$

The implicit reward is $\hat r(x,y)=\beta\log\frac{\pi_\theta(y\mid x)}{\pi_{\text{ref}}(y\mid x)}$. DPO is simpler, stable, and cheap → it became the default for many open models. Tradeoff: it's offline (fixed preference set), so it can't explore as freely as online PPO; the reference-KL is baked in via $\beta$.

DPO loss vs the reward margin $\beta(\hat r_w-\hat r_l)$: it's a logistic loss pushing the chosen response's log-prob above the rejected one. Large positive margin → near-zero loss; negative margin (model prefers the worse answer) → large gradient.

◆ Frontier trick — the post-training landscape, with opinions
  • Reward hacking is the central failure. Symptoms: rising reward, falling human quality. Causes: reward-model overfitting/off-distribution, length bias (RMs love longer answers), sycophancy. Mitigations: KL leash, length normalization/penalty, RM ensembles, fresh on-policy preference data, periodic re-collection.
  • GRPO (used for reasoning models): a PPO variant that drops the value/critic network and estimates advantage from a group of sampled completions' rewards (normalized within the group). Cheaper, stable — powering the recent reasoning-RL wave.
  • RLVR (verifiable rewards): for math/code, skip the learned RM — reward = "did the unit tests pass / is the answer correct." Clean, unhackable signal; the engine behind reasoning gains and long chains-of-thought / test-time compute.
  • RLAIF / Constitutional AI: replace human labels with an AI judge guided by a written constitution → scales preference data cheaply.
  • Online > offline when you can afford it: PPO/GRPO sample on-policy and keep improving; DPO is offline and can stale. Many labs do DPO-style for cost, online RL for the frontier.
  • SFT → preference-opt → (optionally) verifiable-RL is the canonical recipe; each stage targets a different gap.

Debug drill — "garbage after the first generated token"

🎤 Mock interview (AI-coding round) A decoder-only model scores fine on teacher-forced perplexity and the first generated token is always correct, but from the second token onward the generation is repetitive nonsense. Greedy and sampling both break. The KV cache was just added for speed. Find the bug, explain why only generation (not training) is affected, and give the fix. 
def decode_step(model, new_token, cache, step_idx):
    # new_token: the single token id just produced, shape [B, 1]
    x = model.embed(new_token)                       # [B, 1, d]
    # RoPE positions for this step:
    pos = torch.arange(x.size(1), device=x.device)   # <-- [0]  every step
    x = apply_rope(x, pos)
    for layer in model.layers:
        q, k, v = layer.qkv(x).chunk(3, dim=-1)
        cache[layer].append(k, v)                    # append new k,v
        k_all, v_all = cache[layer].get()            # all past + current
        x = layer.attn(q, k_all, v_all) + x
        x = layer.mlp(layer.norm(x)) + x
    return model.head(x)

Try it before revealing. (1) What position does RoPE assign to the new token each step? (2) Why is the first token fine but the rest garbage? (3) Why didn't training catch it? (4) The fix.

Reveal the answer▶
The bug: pos = torch.arange(x.size(1)) is [0] on every decode step, because x is always length 1. So RoPE rotates every generated token as if it were at position 0. The query's relative position to the cached keys is therefore wrong for all steps after the first.

Why the first token is fine: at the first decode step the cache holds the prompt and the one new query at position 0 lines up acceptably; but once the cache holds tokens at positions 0,1,2,… while every new query keeps being rotated as position 0, the relative offsets $n-m$ are all corrupted → attention lands on the wrong positions → repetition/nonsense that compounds.

Why training didn't catch it: training is teacher-forced over the full sequence in one forward pass, where positions are arange(seq_len) correctly — the buggy single-token path only runs at incremental decode. Perplexity (computed the training way) looks perfect. Classic "passes eval, fails generation" KV-cache bug.

The fix — feed the running absolute position (the cache length so far): 
pos = torch.arange(step_idx, step_idx + x.size(1), device=x.device)
x = apply_rope(x, pos)   # token at its true absolute position step_idx
One-liner: "Incremental decode must pass the absolute position offset to RoPE (and the right causal slice to attention); resetting position to 0 each step corrupts every relative offset after the first token. Training is full-sequence so it never exercises this path — always test free-running generation, not just teacher-forced loss."

What top-p does: keep the smallest set of tokens whose cumulative probability reaches $p$ (the "nucleus"), zero the rest, renormalize, sample. ★ Drill 5 in depth — fp16 range, and how loss scaling rescues it

It's a range problem, not precision. fp16 has only 5 exponent bits, so it can represent magnitudes in roughly $[6\times10^{-5},\,6.5\times10^{4}]$. Anything below the floor underflows to 0; anything above the ceiling overflows to inf (and then $0/0$ or $\inf-\inf \to$ NaN).

  • Gradients have a long small-magnitude tail that sits below fp16's floor → they round to 0 → those weights stop learning, and the bad values can cascade to NaN.
  • bf16 keeps fp32's 8 exponent bits (range ±3.4e38) — only the mantissa is smaller — so it almost never over/underflows. That's the whole reason "bf16 fine, fp16 NaN."

Loss scaling fixes fp16 by shifting the gradient distribution up into the representable window: multiply the loss by a large $S$ before backward() (by linearity every gradient is scaled by $S$), then divide by $S$ before the optimizer step:

$$\mathcal{L}' = S\cdot\mathcal{L}\ \Rightarrow\ g' = S\cdot g\ \ (\text{lifted into range}),\qquad \theta \leftarrow \theta - \eta\,\tfrac{1}{S}g'$$

A dynamic scaler (PyTorch GradScaler) auto-tunes $S$: grow it when stable, halve it (and skip the step) on an inf/NaN — the sawtooth from Day 1 §6.3. You also keep an fp32 master copy of weights (a tiny update added to an fp16 weight would round away) and keep softmax/LayerNorm/loss in fp32.

Gradient magnitudes vs the fp16 window (shaded region is representable; left of it underflows to 0). The raw gradient distribution (orange) has a fat tail below the floor → underflow → NaN. Scaling by $S$ (blue) lifts the whole distribution inside the window. bf16 needs none of this because its range already covers the tail.

★ Drill 6 in depth — the block-diagonal ("document") mask, visualized

Why pack at all? Training is most efficient when every sequence is full length $L$. Real documents vary in length, so you concatenate several into one packed sequence to avoid padding waste. The catch: attention and positions must respect the hidden document boundaries.

How the mask diagnoses/fixes it. Attention is governed by the $L\times L$ mask (Day 2 §2.4): cell $(i,j)$ allowed ⇒ query $i$ can read key $j$. A plain causal mask is the whole lower triangle — which includes cells where $i$ is in doc B and $j$ is in doc A. Those off-block cells are cross-document leakage: doc B attends into unrelated doc A. The fix is a block-diagonal causal mask — lower-triangular within each document's block only, everything outside the blocks masked. Equivalently: allowed ⇔ ($j \le i$) and (same document).

A packed sequence of 3 documents (boundaries marked). Left (buggy causal): the whole lower triangle is allowed — the red cells are cross-document leakage (later docs reading earlier ones). Right (block-diagonal): only the per-document lower-triangular blocks (green) are allowed; the red leakage is masked out. Note position ids also reset to 0 at each block.

The other half — positions. pos_ids = arange(L) makes doc B's first token think it's at position $l_a$, not 0 — wrong absolute/RoPE positions, and it implies a (masked-out) relationship to doc A. Reset positions per document: 0..la-1, 0..lb-1, …. In practice you pass cumulative sequence lengths (cu_seqlens) to FlashAttention's varlen mode, which builds the block-diagonal mask and per-segment positions for you — no $L\times L$ matrix ever materialized.

Why it "hurts quality" rather than crashes: nothing errors — the model just learns spurious dependencies (a token predicting across a boundary it shouldn't see) and slightly wrong positions, which quietly degrades generalization. The tell: quality dips exactly when you turned on packing.

◆ Frontier trick — cu_seqlens + FlashAttention varlen, unpacked

What cu_seqlens is: the cumulative sequence lengths — a prefix-sum of the packed documents' lengths, with a leading 0. It's the index map of where each document starts and ends in the packed buffer:

What the kernel does with it. You hand flash_attn_varlen_func the packed, unpadded Q/K/V ([total_tokens, heads, head_dim]) plus cu_seqlens and max_seqlen. FlashAttention is already tiled with online softmax (§2.5) and never builds the $L\times L$ matrix; in varlen mode it only iterates over tiles that lie inside a single [cu_seqlens[i], cu_seqlens[i+1]) segment. So:

  • The block-diagonal structure comes from which tiles it computes, not from masking a big matrix. The cross-document region (the red leakage cells above) is never computed at all — skipped, not computed-then-masked.
  • It resets positions per segment from the same cu_seqlens (RoPE 0..len-1 within each doc).

Two wins: (1) memory stays $O(\text{total tokens})$, never $O(L^2)$; (2) compute — a naïve block-diagonal mask still computes all $L^2$ scores then masks half; varlen skips them. For $m$ docs of length $s$ ($L=ms$): full $\propto L^2=m^2s^2$, varlen $\propto m\,s^2=L^2/m$ → about $m\times$ cheaper. Packing many short docs is exactly where this pays off.

Attention compute for a fixed total length $L=4096$ as you pack it into more documents. Naïve causal/masked (orange) stays at full $L^2$ regardless. varlen (green) only computes the per-document blocks → cost falls $\propto 1/m$ (sum of per-doc squares). At 16 packed docs it's ~16× cheaper — and correct (no leakage).

Drill 7 — "multi-GPU loss doesn't match single-GPU"

# DDP, same total batch size as the single-GPU run. Each rank:
loss = cross_entropy(logits, labels, ignore_index=-100, reduction='mean')
loss.backward()        # DDP all-reduces (averages) gradients across ranks

Same global batch size, yet loss/convergence differs. The subtle bug?

Reveal the answer▶
Bug: reduction='mean' averages over each rank's own non-masked token count; DDP then averages those per-rank means equally. With variable padding/masking, ranks have different valid-token counts, so tokens on a rank with fewer valid tokens get over-weighted — not the same as one global mean.

Fix: sum the per-token loss locally, all-reduce the global non-masked token count, divide by that (token-weighted mean). Same care applies to gradient accumulation (divide by the right number of micro-steps).

Trap: "same global batch size ⇒ identical loss" — only true if the reduction denominator is the global token count. (Echoes §3.3.)
★ Drill 7 in depth — what DDP is, and how to write the loss correctly

First, the distributed vocabulary (MPI / collective-communications terms):

  • rank — one process in the distributed job, almost always one GPU. Unique integer ID 0 … world_size−1 ("rank 3" = the 4th GPU process).
  • world_size — total number of processes/GPUs in the job. This is the $R$ in the math below.
  • local_rank — the GPU index within a single machine/node (vs the global rank across all nodes).
  • rank 0 — conventionally the "main" process: does logging, checkpoint saving, etc.
  • collectives (all_reduce, all_gather, broadcast) operate across all ranks at once — every rank contributes and receives the combined result.

So "equal tokens per rank" = each GPU process gets the same number of valid (non-masked) tokens in its batch shard.

What is DDP (DistributedDataParallel)? The standard data-parallel training method. The full model is replicated on every GPU ("rank"); each rank gets a different shard of the batch, runs its own forward + backward to get local gradients, and then DDP performs an all-reduce that averages the gradients across all ranks (every replica ends with the same averaged gradient) so the replicas stay in lockstep after the identical optimizer step. (Contrast: FSDP/ZeRO shard the model + optimizer state instead of replicating — Day 1 §5.5 — but the gradient-averaging point below is the same.)

The exact mismatch. DDP averages gradients: $\bar g=\frac1R\sum_{r} g_r$ (R = world size). With reduction='mean', each rank's loss is $L_r=\frac{1}{t_r}\sum_{t\in r}\ell_t$ where $t_r$ = that rank's valid (non-masked) token count. So token $t$ on rank $r$ ends up with weight

$$w_t=\frac{1}{R\,t_r}\qquad\text{vs. the single-GPU global mean weight}\qquad w_t^{\star}=\frac{1}{T},\ \ T=\sum_r t_r$$

These match only if every $t_r=T/R$ (equal tokens per rank). With variable lengths / padding / loss masking the $t_r$ differ, so tokens on a rank with fewer valid tokens get up-weighted (smaller $t_r$ → bigger $1/(R\,t_r)$) — a different objective than the single-GPU run.

How to write the loss correctly — normalize by the global token count, and cancel DDP's $\div R$:

logits = model(x)
# SUM of per-token losses on this rank (not mean), and the local valid-token count
local_loss = F.cross_entropy(logits.view(-1, V), labels.view(-1),
                             ignore_index=-100, reduction='sum')
local_tok  = (labels != -100).sum()

# global valid-token count across ALL ranks
global_tok = local_tok.clone()
dist.all_reduce(global_tok, op=dist.ReduceOp.SUM)

# divide by GLOBAL tokens; ×world_size cancels DDP's gradient averaging (÷R)
loss = local_loss * dist.get_world_size() / global_tok
loss.backward()        # DDP averages grads → net weight per token = 1/global_tok ✓

Now every token has weight $\frac{1}{R}\cdot\frac{R}{T}=\frac1T$ — exactly the single-GPU global mean. Simpler alternative: make every rank carry the same number of valid tokens (pad/pack to a fixed token budget per rank); then plain reduction='mean' is already correct. Same care for gradient accumulation: divide by the total tokens across all micro-steps, not per-micro-step means.

Two ranks with unequal valid-token counts (Rank A = 100, Rank B = 20). Buggy (mean-of-means): each Rank-B token is weighted ~5× a Rank-A token (orange) — the loss is skewed toward the short-sequence rank. Correct (global token mean): every token weighted equally (green), matching single-GPU.

Part B — rapid-fire concept drills

8. A 70B model serving 100K-token context at batch 32 — the KV cache alone exceeds 100 GB. Why, and how do you cut it?▶

Why: KV bytes = 2 · layers · context · kv_heads · head_dim · bytes · batch — linear in both context and batch, over all layers, so it dwarfs the weights.

Cuts (most leverage first): GQA/MLA (fewer/compressed KV heads, 4–8×+), KV quantization (fp16→int8/int4), PagedAttention (kill fragmentation, share prefixes), sliding-window + attention sinks (bound cache length), shorter context / chunked prefill.

Trap: reaching for more GPUs before shrinking the per-token cache — at decode the cache, not compute, is the binding constraint.

9. RLHF/DPO uses KL(policy ‖ reference). Why that direction, and what changes if you flip it?▶

Answer: it's the reverse / mode-seeking KL — the expectation is under the policy, so you can estimate it from the samples you're already generating. It penalizes the policy putting mass where the reference doesn't → keeps it on the reference's support (no drift into reward-gaming gibberish).

Flipped (forward KL, mode-covering) needs reference samples and pushes the policy to cover all of the reference's modes — wrong objective and hard to estimate on-policy.

Trap: saying "KL is symmetric." Direction = mode-seeking vs mode-covering.

10. Open-ended generations are repetitive and bland with beam search but fine with sampling. Why does beam fail?▶

Answer: beam search maximizes sequence likelihood, and for open-ended text the highest-likelihood continuations are generic/repetitive filler — human text is not the argmax of the model's distribution (the "likelihood trap"). Beam also amplifies repetition loops.

When beam IS right: low-entropy tasks with one correct answer — translation, ASR, constrained decoding.

Trap: treating beam as universally "better decoding." Use top-p/temperature for creative tasks.

★ Drill 10 in depth — the likelihood trap, with examples & tips

What beam does: keep the top-$B$ partial sequences by cumulative log-prob and approximately find the sequence that maximizes total likelihood $\prod_t p(x_t\mid x_{false for open-ended text.

Why human text is not the argmax (Holtzman et al. 2020):

  • Language is high-entropy. At most positions there are many good continuations, so each specific good token gets only modest probability. The single most-likely token is the safest/blandest (a function word, or "more of the same").
  • Maximizing the product compounds blandness → the argmax sequence has near-zero surprise per token. But human writing keeps roughly uniform, moderate information density (every token carries some surprise), so real text sits at moderate probability, not the peak.
  • Repetition is self-reinforcing: once a phrase repeats, the model rates the repeat even higher → a likelihood-maximizer falls into and deepens the loop ("neural text degeneration").

Examples: "My favorite hobby is" → greedy/beam: "reading. My favorite hobby is reading. My favorite hobby is reading…" (loop) or flat filler. Degenerate extremes like "the the the". Versus translation ("the cat sat" → ~one correct French sentence): beam is great because the target is near the mode.

The canonical picture (Holtzman-style): per-token probability along a generation. Beam/greedy text stays suspiciously high and flat (near 1.0) — low surprise, the tell of degeneration. Human text is jagged and moderate — it dips on surprising/content words. Maximizing likelihood chases the flat high line; good text lives in the jagged band.

◆ Frontier tips — decoding strategy by task
  • Match decoder to task: sampling (top-p / temperature / top-k, newer min-p) for open-ended/creative; greedy/beam for closed-ended (translation, ASR, structured extraction, code-with-verifier).
  • Patches if you must reduce loops: repetition penalty, no-repeat-ngram, frequency/presence penalties — band-aids over the likelihood trap, not a real fix.
  • Reasoning models split exploit vs explore: low-temp/greedy for the final answer (deterministic, correct), but sample many chains for self-consistency (majority vote) and best-of-N / pass@k. Decoding choice = exploration vs exploitation.
  • Beam mostly left LLM chat — still standard in NMT/ASR. Newer alternatives: contrastive search / contrastive decoding penalize degeneration while staying coherent.
  • Don't confuse with speculative decoding (Day 2 §4) — that's a speed trick that preserves the sampling distribution exactly, not a different strategy.
11. A model is strong in English but weak and ~3× more expensive per request in Hindi. Diagnose.▶

Answer: tokenizer fertility. A vocab trained mostly on English fragments other scripts into many more tokens, so the same sentence costs 2–4× tokens → shorter usable context, higher latency/price, fewer training updates per concept. It's a tokenization + data-mix problem before a modeling one.

Fixes: measure tokens-per-character per language; rebalance the tokenizer corpus / expand vocab coverage; up-weight the language in pretraining.

Trap: adding Hindi fine-tuning data without noticing the tokenizer makes every Hindi example structurally more expensive and lower-resolution.

12. During RLHF the reward keeps rising but human-rated quality drops; outputs get longer and sycophantic. What's happening, and the fix?▶

Answer: reward hacking — the policy exploits reward-model weaknesses instead of improving. Common modes: length bias (RMs over-reward long answers), sycophancy, and drift off the RM's training distribution where its scores are unreliable.

Fixes: stronger KL leash, length normalization/penalty, RM ensembles/uncertainty, fresh on-policy preference data (re-collect as the policy moves), and verifiable rewards (RLVR) where possible.

Confirming metric / trap: reward up while a held-out human eval (or separate unhacked judge) drops — never trust the reward curve as ground truth.
