1.1 Greedy decoding & the quality–diversity tradeoff

At each step a language model outputs logits $z\in\mathbb{R}^{|V|}$; softmax turns them into a distribution $p_i=e^{z_i}/\sum_j e^{z_j}$, and a decoding rule picks the next token. The simplest is greedy — always take the argmax:

import torch, torch.nn.functional as F

def greedy(logits):                 # logits: [vocab]
    return int(logits.argmax(-1))   # most-likely token, deterministic

Greedy is coherent but repetitive and bland, and it gets stuck in loops (degeneration). The fundamental tension in all decoding: quality/coherence ↔ diversity. Every method below is a way to inject controlled randomness — reshape or truncate the distribution — to get diversity without sampling incoherent tail tokens.

1.2 Temperature

Temperature $T$ rescales logits before softmax: $p_i(T)=\dfrac{e^{z_i/T}}{\sum_j e^{z_j/T}}$. $T\to0$ → greedy (peaky); $T=1$ → the model's native distribution; $T\to\infty$ → uniform. $T\lt1$ sharpens, $T\gt1$ flattens.

def sample_with_temperature(logits, T=1.0):
    probs = F.softmax(logits / T, dim=-1)     # divide the LOGITS, not the probs
    return int(torch.multinomial(probs, 1))

The same five logits at three temperatures: $T=0.5$ concentrates mass on the top token (safe), $T=1.5$ spreads it toward the tail (diverse, riskier). Temperature reshapes the whole distribution but never removes the tail — that's what top-k/top-p add.

1.3 Top-k sampling

Keep only the $k$ highest-probability tokens, set the rest to $-\infty$, renormalize, sample. Simple, but $k$ is a fixed count — too generous when the model is confident, too strict when it's uncertain.

def top_k_filter(logits, k):
    kth = torch.topk(logits, k).values[..., -1, None]   # the k-th largest logit
    return logits.masked_fill(logits < kth, -float('inf'))

1.4 Top-p (nucleus) sampling

Keep the smallest set of tokens whose cumulative probability $\ge p$ (the "nucleus"), then renormalize. The cutoff is adaptive — few tokens when confident, many when uncertain — which is why top-p ($p\approx0.9$–$0.95$) is the open-ended default. The two classic bugs are the off-by-one boundary and forgetting to renormalize:

def top_p_filter(logits, p):
    sorted_logits, sorted_idx = torch.sort(logits, descending=True)
    cum = F.softmax(sorted_logits, dim=-1).cumsum(dim=-1)
    remove = cum > p                       # tokens past the nucleus
    remove[..., 1:] = remove[..., :-1].clone()  # SHIFT: keep the token that crosses p
    remove[..., 0] = False                  # never drop the top token
    remove = remove.scatter(-1, sorted_idx, remove)   # back to original order
    return logits.masked_fill(remove, -float('inf'))
# (softmax over the filtered logits renormalizes automatically)

1.5 Min-p, repetition penalty, and composing them

Min-p keeps tokens with $p_i\ge p_\text{min}\cdot\max_j p_j$ — a threshold relative to the top token, robust at high temperature. Repetition penalty divides the logits of already-used tokens to break loops (CTRL-style).

def min_p_filter(logits, pmin=0.1):
    probs = F.softmax(logits, dim=-1)
    thresh = pmin * probs.max(dim=-1, keepdim=True).values
    return logits.masked_fill(probs < thresh, -float('inf'))

def repetition_penalty(logits, generated_ids, penalty=1.2):
    for t in set(generated_ids):
        logits[t] = logits[t] / penalty if logits[t] > 0 else logits[t] * penalty
    return logits

def sample(logits, T=0.8, k=None, p=0.95):     # the usual composition
    logits = logits / T
    if k: logits = top_k_filter(logits, k)
    if p: logits = top_p_filter(logits, p)
    probs = F.softmax(logits, dim=-1)
    return int(torch.multinomial(probs, 1))
⚙ Defaults Open-ended chat: $T\approx0.7$–$1.0$ + top-p $0.9$–$0.95$. Code/math with one right answer: greedy / very low $T$. RL rollouts & best-of-n: higher $T$ for exploration. Order matters: temperature reshapes, then truncate, then renormalize via softmax.

1.6 Beam search

Sampling picks one token at a time; beam search instead keeps the $B$ highest-probability sequences ("beams") at every step, approximately maximizing total log-probability $\sum_t\log p(y_t\mid y_{\lt t})$. It's the right tool for low-entropy tasks (translation, ASR) where one high-probability sequence is correct — and the wrong tool for open-ended chat (it produces bland, repetitive "safe" text and collapses diversity).

import math
def beam_search(step_logprobs, start, eos, B=4, max_len=40):
    # step_logprobs(seq) returns the next-token log-prob vector [vocab]
    beams = [(start, 0.0)]                      # (sequence, cumulative log-prob)
    for _ in range(max_len):
        cands = []
        for seq, score in beams:
            if seq[-1] == eos:                  # finished beam: carry forward
                cands.append((seq, score)); continue
            lp = step_logprobs(seq)             # log-probs of next token
            topv, topi = torch.topk(lp, B)
            for v, t in zip(topv.tolist(), topi.tolist()):
                cands.append((seq + [t], score + v))   # extend, accumulate log-prob
        beams = sorted(cands, key=lambda sc: sc[1], reverse=True)[:B]   # keep best B
        if all(s[-1] == eos for s, _ in beams): break
    # length-normalize so long sequences aren't unfairly penalized
    return max(beams, key=lambda sc: sc[1] / len(sc[0]))[0]

Beam width $B=2$: every beam is expanded by all next tokens, the $2$ best partial sequences (by cumulative log-prob) survive, the rest are pruned. Larger $B$ = better search but more compute and less diversity (beams share prefixes). Length normalization (divide by length) stops it from preferring short sequences.

◆ Why beam search is wrong for chat Open-ended text is high-entropy: the single most-likely sequence is generic, repetitive, and "safe" (the likelihood-degeneration paradox). Beam also collapses diversity and is slow. So chatbots sample (top-p); beam stays for tasks with a sharply peaked correct answer (translation/ASR).

1.7 Sampling drills

Mix of conceptual and code-debug. Answer out loud, then reveal.

D1 — Temperature vs top-p: what does each control, and why combine?▶
Temperature reshapes the whole distribution (sharpen/flatten) but never removes the tail; top-p truncates to an adaptive nucleus (smallest set with cumulative $\ge p$) then renormalizes. Combine: temperature sets overall randomness, top-p cuts the unreliable tail so high-$T$ diversity doesn't produce garbage.
D2 — Implement top-p. What are the two classic bugs?▶
Sort descending, cumsum the softmax, mark cum > p for removal — then (bug 1) shift the mask right by one so you keep the token that crosses $p$ (else if the top token already exceeds $p$ you remove everything → crash). Bug 2: renormalize after truncation before sampling (softmax over the filtered logits does this). See §1.4 code.
D3 — Why is beam search bad for open-ended generation?▶
High-entropy text: the single most-likely sequence is bland/repetitive (likelihood degeneration); beam collapses diversity (shared prefixes) and is slow. Sampling with top-p restores human-like variety. Keep beam for low-entropy tasks with a peaked correct sequence.
D4 — Code-debug: this "temperature" is wrong. Why?
probs = softmax(logits); probs = probs ** (1/T); probs /= probs.sum()▶
Temperature scales the logits before softmax: $\text{softmax}(z/T)$. Exponentiating the probabilities is a different ad-hoc sharpening that doesn't equal the temperature softmax (and drops numerical stability). Fix: probs = softmax(logits / T).
D5 — Greedy keeps repeating "the the the". Two fixes?▶
Degeneration. (1) Switch to sampling (top-p) — greedy/argmax of a high-likelihood model loops. (2) Add a repetition/frequency penalty that lowers logits of already-used tokens. Beam alone won't fix it (it also maximizes likelihood).

2.1 The MDP — the language of RL

RL studies an agent acting in an environment to maximize cumulative reward. The environment is a Markov Decision Process $(S,A,P,R,\gamma)$:

  • States $S$, actions $A$.
  • Transitions $P(s'\mid s,a)$ — probability of next state (the "model"/dynamics).
  • Reward $R(s,a)$, discount $\gamma\in[0,1)$.
  • Policy $\pi(a\mid s)$ — the agent's behavior; return $G_t=\sum_{k\ge0}\gamma^k r_{t+k}$.

Markov property: the next state depends only on the current state and action, not the full history — which is what makes the value recursions below valid. The goal: find a policy maximizing expected return.

2.2 Value functions & the Bellman equations

Two value functions measure "how good":

$$V^\pi(s)=\mathbb{E}_\pi[G_t\mid s_t=s],\qquad Q^\pi(s,a)=\mathbb{E}_\pi[G_t\mid s_t=s,a_t=a].$$

Each satisfies a Bellman equation (value = immediate reward + discounted future value), and the optimal values satisfy the Bellman optimality equations (with a max over actions):

$$Q^\pi(s,a)=\mathbb{E}\big[r+\gamma\,V^\pi(s')\big],\qquad Q^\star(s,a)=\mathbb{E}\big[r+\gamma\max_{a'}Q^\star(s',a')\big].$$

Two ideas that name every algorithm Bootstrapping = update an estimate using other estimates (the $\gamma V(s')$ term) instead of waiting for the true return — this is what TD/DP do and Monte-Carlo does not. Sampling = learn from experienced transitions instead of a known model — this is what MC/TD do and DP does not. The whole classical taxonomy is just these two axes (§2.7).

2.3 On-policy vs off-policy — the concept to nail

Two policies are in play whenever an agent learns:

  • Behavior policy — the one that generates the data (chooses actions while interacting). It usually explores (e.g. $\varepsilon$-greedy).
  • Target policy — the one you're learning about / improving (often the greedy optimal policy).

On-policy: behavior = target. You learn about the same policy you act with (so it must keep exploring, and you can't reuse old data after the policy changes). Off-policy: behavior ≠ target. You learn about a different policy than the one generating data — letting you learn the optimal policy from exploratory or even old/replayed data.

Tie to Day 6 (LLMs) PPO/REINFORCE are on-policy — they must sample from the current model, which is why old samples go stale (the importance ratio / clip exists to allow a little reuse). DPO is off-policy / offline — it learns from a fixed preference dataset not generated by the current policy, which is exactly why it can't explore. "Online vs offline" and "on- vs off-policy" are the same axis showing up in LLM post-training.

2.4 Dynamic programming — when you know the model

If you know $P$ and $R$, you can compute the optimal policy exactly by iterating the Bellman optimality equation. Value iteration:

import numpy as np
def value_iteration(P, R, gamma=0.99, tol=1e-6):
    # P[s,a,s2] transition prob, R[s,a] reward; S states, A actions
    S, A, _ = P.shape
    V = np.zeros(S)
    while True:
        Q = R + gamma * np.einsum('sap,p->sa', P, V)   # one Bellman backup, all (s,a)
        V_new = Q.max(axis=1)                          # greedy over actions
        if np.abs(V_new - V).max() < tol: break        # converged
        V = V_new
    policy = Q.argmax(axis=1)                           # greedy policy
    return V, policy

Policy iteration alternates policy evaluation (solve $V^\pi$ for the current $\pi$) and policy improvement (act greedily w.r.t. $V^\pi$) until the policy stops changing — often fewer iterations than value iteration. Both are bootstrapping, no sampling (they use the known model).

2.5 Model-free prediction — Monte Carlo & TD

Usually you don't know the model — you learn from experience. To estimate $V^\pi$:

  • Monte Carlo (MC): run a full episode, then set each visited state's value toward the actual return $G$. Unbiased, but high variance and needs episodes to finish.
  • Temporal Difference TD(0): update after each step using a bootstrapped target $r+\gamma V(s')$. Lower variance, biased (depends on current $V$), works online and on continuing tasks.
# Monte Carlo prediction (every-visit)
def mc_predict(episodes, gamma=0.99):
    V, counts = defaultdict(float), defaultdict(int)
    for episode in episodes:                 # episode = [(s, r), ...]
        G = 0.0
        for s, r in reversed(episode):       # accumulate return backwards
            G = r + gamma * G
            counts[s] += 1
            V[s] += (G - V[s]) / counts[s]   # running mean of returns
    return V

# TD(0): update every step toward r + gamma*V(s')
def td0_update(V, s, r, s2, alpha=0.1, gamma=0.99):
    V[s] += alpha * (r + gamma * V[s2] - V[s])    # the TD error in brackets
    return V
⚙ The bias–variance line to say MC = unbiased, high variance (the full return is noisy). TD = biased (bootstraps off an imperfect $V$), low variance, and can learn online before an episode ends. $n$-step returns and TD($\lambda$) interpolate between them — the same bias–variance knob as GAE in Day 6.

2.6 Model-free control — SARSA vs Q-learning

For control (finding the best policy) we learn $Q(s,a)$ and act $\varepsilon$-greedily. The two canonical algorithms differ in one line — and that line is the on/off-policy distinction:

def eps_greedy(Q, s, eps, A):
    return np.random.randint(A) if np.random.rand() < eps else int(Q[s].argmax())

# SARSA — ON-policy: target uses the action actually taken next (a2 from the SAME policy)
def sarsa(env, episodes, alpha=0.1, gamma=0.99, eps=0.1):
    Q = np.zeros((env.S, env.A))
    for _ in range(episodes):
        s = env.reset(); a = eps_greedy(Q, s, eps, env.A)
        done = False
        while not done:
            s2, r, done = env.step(a)
            a2 = eps_greedy(Q, s2, eps, env.A)            # next action from behavior policy
            Q[s, a] += alpha * (r + gamma * Q[s2, a2] - Q[s, a])   # uses Q[s2, a2]
            s, a = s2, a2
    return Q

# Q-LEARNING — OFF-policy: target uses the greedy max (the optimal target policy)
def q_learning(env, episodes, alpha=0.1, gamma=0.99, eps=0.1):
    Q = np.zeros((env.S, env.A))
    for _ in range(episodes):
        s = env.reset(); done = False
        while not done:
            a = eps_greedy(Q, s, eps, env.A)              # behavior explores
            s2, r, done = env.step(a)
            Q[s, a] += alpha * (r + gamma * Q[s2].max() - Q[s, a])  # uses max_a' Q[s2,a']
            s = s2
    return Q
The one-line difference (a classic interview question) SARSA's target is $r+\gamma\,Q(s',a')$ where $a'$ is the action the behavior policy actually takes next → it learns the value of the policy it follows (exploration included) → on-policy. Q-learning's target is $r+\gamma\max_{a'}Q(s',a')$ → it learns the value of the greedy/optimal policy regardless of how it explores → off-policy. Consequence: near a cliff, SARSA learns a safer path (it accounts for its own $\varepsilon$-exploration occasionally stepping off), Q-learning learns the optimal (riskier) path. Same update, one symbol — Q[s2,a2] vs Q[s2].max().

2.7 The whole classical map (two axes)

Every method is a point on two axes: do you bootstrap? (use estimates of future value) and do you sample? (learn from experience vs a known model).

No bootstrap (full return)Bootstrap (use $V/Q$ estimate)
Known model (no sampling)exhaustive tree searchDP: value / policy iteration
Learn from experience (sampling)Monte CarloTD: TD(0), SARSA, Q-learning
  • Prediction (evaluate a fixed $\pi$) vs control (find the best $\pi$): MC/TD do prediction; SARSA/Q-learning add $\varepsilon$-greedy improvement for control.
  • On- vs off-policy: SARSA on, Q-learning off (the §2.6 line).
  • Tabular → function approximation: replace the $Q$ table with a neural net and Q-learning becomes DQN (+ replay buffer, target network); policy-gradient on the actor side gives REINFORCE → A2C → PPO (Day 6).

2.8 RL drills

Conceptual + code-debug. Out loud, then reveal.

D6 — Define on-policy vs off-policy in one sentence each, with an example.▶
On-policy: learn about the same policy you use to act (behavior = target) — e.g. SARSA, PPO; must keep exploring, can't reuse stale data. Off-policy: learn about a different (target) policy than the behavior policy generating data — e.g. Q-learning/DQN; enables replay and learning the optimal policy from exploratory data.
D7 — SARSA vs Q-learning: the exact difference and a consequence.▶
SARSA target $r+\gamma Q(s',a')$ uses the action actually taken (on-policy); Q-learning target $r+\gamma\max_{a'}Q(s',a')$ uses the greedy max (off-policy). Consequence (cliff-walking): SARSA learns a safer path because it accounts for its own exploration stepping off the cliff; Q-learning learns the optimal but riskier path. One symbol: Q[s2,a2] vs Q[s2].max().
D8 — Value iteration vs policy iteration?▶
Both need the model and use the Bellman equation. Value iteration: repeatedly apply the Bellman optimality backup (max over actions) until $V$ converges, then read off the greedy policy. Policy iteration: alternate full policy evaluation (solve $V^\pi$) and policy improvement (greedy) — usually fewer outer iterations but each is costlier. Both converge to the optimal policy.
D9 — MC vs TD: bias/variance and when each shines?▶
MC uses the full return → unbiased but high variance, needs complete episodes. TD bootstraps ($r+\gamma V(s')$) → biased (depends on current $V$) but low variance, learns online and on continuing tasks. $n$-step / TD($\lambda$) interpolate. TD usually learns faster in practice; MC is robust when the value function is hard to bootstrap.
D10 — Why do you need exploration (ε-greedy)? What if ε=0?▶
Without exploration the agent only ever tries actions it currently thinks are best → it can lock onto a suboptimal action and never discover better ones (no data on alternatives). $\varepsilon$-greedy takes a random action with prob $\varepsilon$ to keep gathering information. $\varepsilon=0$ → pure exploitation → can converge to a suboptimal policy; usually anneal $\varepsilon$ from high to low.
D11 — Code-debug: this Q-learning update is off-policy in name only. Fix it.
a2 = eps_greedy(Q, s2, eps); Q[s,a] += alpha*(r + gamma*Q[s2,a2] - Q[s,a])▶
That's actually SARSA (it bootstraps off the sampled next action $a2$). For Q-learning (off-policy) the target must use the greedy max, independent of the behavior policy: Q[s,a] += alpha*(r + gamma*Q[s2].max() - Q[s,a]). Using Q[s2,a2] makes it on-policy SARSA.
D12 — Code-debug: TD update with a sign error.
V[s] += alpha * (r + gamma*V[s2] + V[s])▶
The TD error is $\delta=r+\gamma V(s')-V(s)$ — you subtract the current estimate, not add it: V[s] += alpha * (r + gamma*V[s2] - V[s]). With + V[s] the update diverges (it's no longer moving $V(s)$ toward its target). The bracket must be the prediction error.
D13 — How does Q-learning become DQN?▶
Replace the tabular $Q$ with a neural net $Q_\theta(s,a)$ and regress toward the bootstrapped target. Two tricks make it stable: a replay buffer (store transitions, sample minibatches → off-policy reuse + decorrelation) and a target network (a slow copy for the $\max$ target → avoids chasing a moving target). Off-policy is what makes replay valid.
D14 — Connect classical RL to LLM post-training (Day 6).▶
REINFORCE = policy gradient (on-policy), the actor side; add a learned value/critic → actor-critic/A2C → add a trust-region clip → PPO (on-policy, must sample from the current model). DPO is off-policy/offline (fixed preference data). GRPO is REINFORCE-with-a-group-baseline (critic-free). The on/off-policy, bootstrap, and baseline ideas here are exactly the ones scaled up for LLMs.
